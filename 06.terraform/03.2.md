# Module 3 – Complete Hands-On Project (Remote Backend + Multi-Environment Setup)

This guide explains FROM SCRATCH how to:

* Create a proper Terraform project structure
* Configure S3 remote backend
* Enable DynamoDB state locking
* Separate dev / test / prod environments
* Apply infrastructure safely per environment

---

# STEP 1 – Install & Configure

Ensure:

* Terraform installed
* AWS CLI installed
* AWS credentials configured

Verify:

```
terraform -version
aws sts get-caller-identity
```

---

# STEP 2 – Create Project Structure

Create folders:

```
terraform-project/
│
├── backend/
│   └── backend.tf
│
├── environments/
│   ├── dev/
│   │   └── terraform.tfvars
│   ├── test/
│   │   └── terraform.tfvars
│   └── prod/
│       └── terraform.tfvars
│
├── main.tf
├── variables.tf
├── outputs.tf
└── providers.tf
```

This structure allows one codebase with multiple environments.

---

# STEP 3 – Create S3 Bucket (One-Time Setup)

Create S3 bucket manually OR using Terraform separately.

Example bucket name:

```
my-terraform-remote-state-bucket
```

Enable:

* Versioning
* Server-side encryption

---

# STEP 4 – Create DynamoDB Lock Table

Create table:

* Table name: terraform-lock-table
* Partition key: LockID (String)
* Billing mode: On-demand

This enables state locking.

---

# STEP 5 – Configure Backend

Create file: backend/backend.tf

```
terraform {
  backend "s3" {
    bucket         = "my-terraform-remote-state-bucket"
    key            = "ENV/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock-table"
    encrypt        = true
  }
}
```

IMPORTANT:
Replace ENV manually per environment during init.

main.tf

```

  count         = var.instance_count
```

# STEP 10 – Initialize Per Environment

For DEV:

```
terraform init \
  -backend-config="key=dev/terraform.tfstate"

terraform apply \
  -var-file="environments/dev/terraform.tfvars"
```

---

For TEST:

```
terraform init \
  -backend-config="key=test/terraform.tfstate"

terraform apply \
  -var-file="environments/test/terraform.tfvars"
```

---

For PROD:

```
terraform init \
  -backend-config="key=prod/terraform.tfstate"

terraform apply \
  -var-file="environments/prod/terraform.tfvars"
```

Each environment now has:

* Separate state file
* Separate infrastructure
* Same codebase

# IMPORTANT: Can We Use Variables Inside Backend Block?

Very common doubt:

Can we write like this?

```hcl
terraform {
  backend "s3" {
    bucket = var.bucket_name   # ❌ This will NOT work
  }
}
```

Answer: ❌ NO.

Why?

Terraform works in two phases:

1. `terraform init` → Backend initializes first
2. `terraform plan/apply` → Variables are loaded later

Backend runs BEFORE variables are evaluated.

So Terraform does not allow `var.*` inside backend block.

---

# How To Handle This Correctly (Simple & Practical)

Instead of using variables, we use one of the following real-world methods.

---

## Method 1 – Partial Backend + CLI Arguments (Very Common)

Keep backend block empty:

```hcl
terraform {
  backend "s3" {}
}
```

Then initialize per environment:

DEV:

```
terraform init \
  -backend-config="bucket=my-terraform-remote-state-bucket" \
  -backend-config="key=dev/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=terraform-lock-table" \
  -backend-config="encrypt=true"
```

TEST:

```
terraform init \
  -backend-config="bucket=my-terraform-remote-state-bucket" \
  -backend-config="key=test/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=terraform-lock-table" \
  -backend-config="encrypt=true"
```

Notice:
Only the `key` changes per environment.

---

## Method 2 – Separate backend.hcl Files (Cleaner Approach)

Create files:

```
backend-dev.hcl
backend-test.hcl
backend-prod.hcl
```

Example: backend-dev.hcl

```
bucket         = "my-terraform-remote-state-bucket"
key            = "dev/terraform.tfstate"
region         = "us-east-1"
dynamodb_table = "terraform-lock-table"
encrypt        = true
```

Initialize:

```
terraform init -backend-config=backend-dev.hcl
```

For prod:

```
terraform init -backend-config=backend-prod.hcl
```

This approach is:

✔ Cleaner
✔ Easier for CI/CD
✔ Safer for production

---

# Final Understanding

Backend configuration:

* Cannot use Terraform variables
* Must be decided during `terraform init`
* Should be separated per environment using key or backend files

This is how real-world multi-environment Terraform projects manage remote state safely.
