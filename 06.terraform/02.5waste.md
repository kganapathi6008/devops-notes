# 1. Input Variables

Input variables allow you to pass values into your Terraform configuration instead of hardcoding them.

They separate:

* Infrastructure logic (inside `.tf` files)
* Environment-specific values (inside `.tfvars` files)

This makes your configuration reusable, clean, and production-ready.

---

## Without Variables (Hardcoded Example)

```hcl
resource "aws_instance" "ec2_instance" {
  ami           = "ami-0f3caa1cf4417e51b"
  instance_type = "t3.micro"
}
```

Problem:

* If you want to change region, AMI, instance type, or tags → you must edit the code.
* Not reusable for multiple environments.

---

## With Input Variables (Dynamic Configuration)

### Step 1: Define Variables (variables.tf)

```hcl
variable "region" {
  type = string
}

variable "ec2_ami_id" {
  type = string
}

variable "ec2_instance_type" {
  type = string
}

variable "key_pair_name" {
  type = string
}

variable "security_group_name" {
  type = string
}

variable "allowed_inbound_cidrs" {
  type = list(string)
}

variable "allowed_outbound_cidrs" {
  type = list(string)
}

variable "resource_tags" {
  type = map(string)
}
```

---

### Step 2: Use Variables in Resources

```hcl
provider "aws" {
  region = var.region
}

resource "aws_instance" "ec2_instance" {
  ami           = var.ec2_ami_id
  instance_type = var.ec2_instance_type
  key_name      = var.key_pair_name

  vpc_security_group_ids = [
    aws_security_group.application_sg.id
  ]

  tags = var.resource_tags
}
```

---

### Step 3: Provide Values Using dev.tfvars

```hcl
region = "us-east-1"

ec2_ami_id        = "ami-0f3caa1cf4417e51b"
ec2_instance_type = "t3.micro"
key_pair_name     = "application-key"

security_group_name = "application-security-group"

allowed_inbound_cidrs = [
  "0.0.0.0/0"
]

allowed_outbound_cidrs = [
  "0.0.0.0/0"
]

resource_tags = {
  Name        = "Application-Server"
  Environment = "dev"
  ManagedBy   = "Terraform"
}
```

---

## How to Run

### Plan

```bash
terraform plan --var-file=dev.tfvars
```

### Apply

```bash
terraform apply --auto-approve --var-file=dev.tfvars
```

---

## What Happened Practically

Using variables, Terraform:

* Created EC2 instance
* Created Security Group (22 & 443)
* Generated SSH key pair
* Saved `.pem` file locally
* Assigned tags dynamically
* Used region from variable

You connected using:

```bash
ssh -i application-key.pem ec2-user@<public-ip>
```

---

## Why Input Variables Are Useful

* Same configuration works for dev, test, prod
* No code modification needed for environment changes
* Clean separation of logic and values
* Easy automation in CI/CD
* Makes modules reusable

---

## Interview-Level Definition

Input variables in Terraform allow externalizing configuration values so infrastructure code remains reusable, environment-agnostic, and maintainable.

# Multiple tfvars Files and State Behavior (Important Concept)

Suppose you have:

* dev.tfvars
* test.tfvars
* prod.tfvars

And you run:

```bash
terraform apply --auto-approve --var-file=dev.tfvars
terraform apply --auto-approve --var-file=test.tfvars
terraform apply --auto-approve --var-file=prod.tfvars
```

## Where Is the State File Stored?

If you are NOT using a remote backend (like S3), Terraform uses the **local backend by default**.

That means state is stored in:

```
terraform.tfstate
```

inside your current working directory.

---

## What Actually Happens

All three applies use the SAME state file:

```
terraform.tfstate
```

So:

* First apply (dev) → State represents DEV
* Second apply (test) → Terraform compares with DEV state and modifies same infrastructure
* Third apply (prod) → Terraform again updates same resources

You are NOT creating three separate environments.

You are modifying the same environment repeatedly.

After the final apply, the state file represents only the LAST applied environment.

---

## Why This Is Important

Without separating state:

* Dev, Test, Prod cannot exist at the same time
* Infrastructure gets replaced or modified
* This is risky in real environments

---

## Safe Ways to Handle Multiple Environments

### Option 1 — Separate Folders (Simple Approach)

```
project/
  dev/
  test/
  prod/
```

Each folder will have its own `terraform.tfstate` file.

---

### Option 2 — Workspaces (Single Folder, Multiple States)

Create workspaces:

```bash
terraform workspace new dev
terraform workspace new test
terraform workspace new prod
```

Switch workspace:

```bash
terraform workspace select dev
terraform apply --var-file=dev.tfvars
```

Terraform stores separate state files under:

```
terraform.tfstate.d/
```

Example:

```
terraform.tfstate.d/dev/terraform.tfstate
terraform.tfstate.d/test/terraform.tfstate
terraform.tfstate.d/prod/terraform.tfstate
```

Now you truly have separate environments.

---

## Interview-Level Explanation

If different tfvars files are applied in the same directory without workspace or backend separation, Terraform uses the same state file and treats it as a single environment. Each apply modifies the existing infrastructure instead of creating independent environments.
