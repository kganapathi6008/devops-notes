# Module 4 – Modules & Environment Design

## 1. What is a Module?

In Terraform, a **module** is a container for multiple resources that are used together.

Think of a module like a **reusable infrastructure template**.

Instead of writing the same infrastructure code again and again, we package it inside a module and reuse it wherever needed.

Example:

* VPC module
* EC2 module
* Security Group module
* RDS module

Benefits:

* Code reusability
* Cleaner architecture
* Easier maintenance
* Standardization across environments

---

## 2. Root Module vs Child Module

### Root Module

The **root module** is the main Terraform configuration where you run:

* terraform init
* terraform plan
* terraform apply

It usually contains:

* provider configuration
* backend configuration
* module calls
* environment specific variables

Example:

```
main.tf (project root)
```

### Child Module

A **child module** is a reusable infrastructure component.

Example modules:

```
modules/
  vpc/
  ec2/
  security-group/
```

These modules contain reusable infrastructure logic.

Example structure:

```
modules/vpc
  main.tf
  variables.tf
  outputs.tf
```

The root module calls these modules.

---

## 3. Module Inputs

Modules accept **inputs using variables**.

In this project, modules receive inputs like **instance type, key name, or security group values**.

Example variable inside module:

```
variable "instance_type" {
  description = "EC2 instance size"
  type        = string
}
```

Root module passes the value to the EC2 module:

```
module "ec2" {
  source = "./modules/ec2"

  instance_type = var.instance_type
}
```

The value of `instance_type` changes depending on the environment.

Example:

**dev.tfvars**

```
instance_type = "t2.micro"
```

**prod.tfvars**

```
instance_type = "t5.large"
```

This allows the same module to be reused across environments with different configurations.

---

## 4. Module Outputs

Modules expose important values using **outputs**.

Example inside module:

```
output "vpc_id" {
  value = aws_vpc.main.id
}
```

Root module can use it:

```
vpc_id = module.vpc.vpc_id
```

Common outputs:

* VPC ID
* Subnet IDs
* Security group IDs
* Instance IDs

Outputs allow modules to **communicate with each other**.

---

## 5. Reusable Architecture Design

In real-world infrastructure we break infrastructure into logical modules.

Example architecture modules:

```
modules/
  vpc/
  security-group/
  ec2/
  rds/
  load-balancer/
```

Each module handles one responsibility.

Example flow:

```
VPC Module
   ↓
Subnet Module
   ↓
Security Group Module
   ↓
EC2 Module
```

This keeps infrastructure **clean, reusable and scalable**.

## 6. Folder Structure Used In This Project

Your project uses a **single root module with environment variable files and backend configs**. This is a common pattern for small to medium Terraform projects.

Project structure:

```
04.1.terraform-modular-updated/

backend/
  backend-dev.hcl
  backend-test.hcl
  backend-prod.hcl

modules/
  ec2/
  ec2-key-pair/
  security-group/

environments/
  dev.tfvars
  test.tfvars
  prod.tfvars

backend.tf
provider.tf
versions.tf
variables.tf
locals.tf
main.tf
outputs.tf
```

Explanation of each part:

### Root Terraform Files

These files exist in the **project root** and form the **Root Module**.

```
main.tf
provider.tf
variables.tf
locals.tf
outputs.tf
backend.tf
versions.tf
```

Purpose of each file:

**main.tf**

Calls all reusable modules such as EC2, Security Group, and Key Pair.

Example:

```
module "ec2" {
  source = "./modules/ec2"
}
```

**provider.tf**

Defines the cloud provider configuration.

Example:

```
provider "aws" {
  region = var.aws_region
}
```

**variables.tf**

Defines all input variables used across the project.

Examples:

* instance type
* VPC ID
* allowed CIDR blocks

**locals.tf**

Used to define reusable calculated values.

Example:

```
locals {
  project_name = "terraform-modular-project"
}
```

**outputs.tf**

Exports useful values such as:

* instance id
* public IP
* security group id

**versions.tf**

Locks Terraform and provider versions.

Example:

```
terraform {
  required_version = ">= 1.0"
}
```

**backend.tf**

Defines the remote backend configuration (S3 state storage).

Actual backend values are provided using `.hcl` files.

---

### Modules Folder

```
modules/
  ec2/
  ec2-key-pair/
  security-group/
```

Each folder represents a **reusable child module**.

Example responsibilities:

**ec2 module**

Creates EC2 instances.

**ec2-key-pair module**

Creates AWS key pairs used for SSH access.

**security-group module**

Creates firewall rules for EC2 instances.

Each module usually contains:

```
main.tf
variables.tf
outputs.tf
```

These modules are called from the **root main.tf**.

---

### Environment Variable Files

```
environments/
  dev.tfvars
  test.tfvars
  prod.tfvars
```

These files store **environment specific variable values**.

Example:

**dev.tfvars**

```
instance_type = "t3.micro"
```

**prod.tfvars**

```
instance_type = "t3.large"
```

During execution we choose the environment:

```
terraform plan -var-file=environments/dev.tfvars
```

This allows the **same Terraform code to deploy different environments**.

---

### Backend Configuration Files

```
backend/
  backend-dev.hcl
  backend-test.hcl
  backend-prod.hcl
```

These files define **where Terraform state is stored** for each environment.

Example:

```
bucket = "my-terraform-state"
key    = "dev/terraform.tfstate"
region = "us-east-1"
```

Initialization example:

```
terraform init -backend-config=backend/backend-dev.hcl
```

This ensures **each environment has its own state file**.

Example states:

```
S3 Bucket

terraform-state
   ├── dev/terraform.tfstate
   ├── test/terraform.tfstate
   └── prod/terraform.tfstate
```

This prevents environments from interfering with each other.

## 7. Dev / Test / Prod Separation

Your project separates environments using **two mechanisms**:

1. Variable files
2. Backend configuration

Example workflow:

**Dev environment**

```
terraform init -backend-config=backend/backend-dev.hcl
terraform plan -var-file=environments/dev.tfvars
terraform apply -var-file=environments/dev.tfvars
```

**Test environment**

```
terraform init -backend-config=backend/backend-test.hcl
terraform plan -var-file=environments/test.tfvars
```

**Production environment**

```
terraform init -backend-config=backend/backend-prod.hcl
terraform plan -var-file=environments/prod.tfvars
```

This approach allows the **same infrastructure code to safely manage multiple environments**.

---

## 8. Separate State Per Environment

Terraform stores infrastructure state inside **terraform.tfstate**.

In real-world systems, each environment must have **separate state files**.

Example:

```
Dev State
s3://terraform-state/dev/terraform.tfstate

QA State
s3://terraform-state/qa/terraform.tfstate

Prod State
s3://terraform-state/prod/terraform.tfstate
```

Why separate state is important:

* Prevent accidental production changes
* Independent infrastructure management
* Safer deployments

Example backend configuration:

```
terraform {
  backend "s3" {}
}
```

Values are usually passed using backend.hcl per environment.

Example:

```
bucket = "my-terraform-state"
key    = "dev/terraform.tfstate"
region = "us-east-1"
```

## 9. Real-World Workflow

Typical Terraform workflow with modules:

```
1. Developer updates module
2. Dev environment tested
3. QA environment deployed
4. Production deployment
```

Commands:

```
terraform init
terraform plan
terraform apply
```

Executed inside the specific environment folder.

Modules help us:
