# Terraform State Refactoring and Resource Renaming – Complete Practical Guide

This document explains in depth what happens when you rename or move Terraform resources, how Terraform interprets those changes, and how to safely refactor infrastructure in real-world environments.

This applies to all Terraform-managed resources, not just security groups.

---

# 1. Core Principle: Terraform Tracks by Resource Address

Terraform does NOT track infrastructure by:

* AWS resource name
* Tags
* ARN alone
* Visible console name

Terraform tracks infrastructure by the **resource address stored in the state file**.

Example resource address:

```
aws_security_group.dev_sg
```

Resource address format:

```
<resource_type>.<resource_label>
```

If you change the resource label in configuration, you change its identity from Terraform’s perspective.

---

# 2. What Actually Happens When You Rename a Resource

Original configuration:

```
resource "aws_security_group" "dev_sg" {
```

State contains:

```
aws_security_group.dev_sg
```

After renaming:

```
resource "aws_security_group" "ec2_security_group" {
```

Terraform compares:

Configuration  vs  State

It sees:

* aws_security_group.dev_sg exists in state but not in configuration
* aws_security_group.ec2_security_group exists in configuration but not in state

Terraform concludes:

* Destroy old resource
* Create new resource

Even though AWS infrastructure did not change.

This behavior is correct and expected.

---

# 3. Why Terraform Cannot Automatically Detect Renames

Terraform is declarative and deterministic.

It does not guess intent.

If Terraform automatically assumed renames, it could:

* Map unrelated resources incorrectly
* Corrupt state
* Hide infrastructure drift
* Cause unintended behavior

Therefore, Terraform requires explicit refactoring instructions.

---

# 4. Real-World Scenarios That Change Resource Address

This commonly happens during:

* Renaming resource labels
* Refactoring naming conventions
* Moving resources into modules
* Moving resources out of modules
* Changing from count to for_each
* Changing for_each keys
* Splitting projects
* Restructuring folders
* Converting monolithic code into modular design

Any change to the resource address requires state migration.

---

# 5. Understanding Complex Resource Addresses

Simple resource:

```
aws_instance.web
```

Using count:

```
aws_instance.web[0]
```

Using for_each:

```
aws_instance.web["app1"]
```

Inside module:

```
module.ec2.aws_instance.web
```

If any part of this address changes, Terraform treats it as a different object.

---

# 6. Safe Refactoring Methods

There are two safe approaches.

---

## Method A: terraform state mv (CLI Method)

Used when renaming or moving resources.

Example:

```
terraform state mv \
  aws_security_group.dev_sg \
  aws_security_group.ec2_security_group
```

What this does:

* Updates state file only
* Does NOT touch AWS
* Prevents destroy and recreate

After running:

```
terraform plan
```

There should be no infrastructure changes.

---

## Method B: moved Block (Declarative Method – Recommended)

Instead of running a CLI command, you can declare the move inside configuration.

Example:

```
moved {
  from = aws_security_group.dev_sg
  to   = aws_security_group.ec2_security_group
}
```

---

### Where to Place the moved Block

The moved block must be placed:

* In the same root module where the resource exists
* In any .tf file within that folder

Example project structure:

```
ec2-project/
  ├── main.tf
  ├── security_group.tf
  ├── variables.tf
  └── versions.tf
```

You can place the moved block in:

* main.tf
* security_group.tf
* Or create a new file like moved.tf

Terraform automatically merges all .tf files in the folder.

---

### What Happens During Apply

Run:

```
terraform plan
```

Terraform will show:

```
Terraform will perform the following actions:

  # aws_security_group.dev_sg has moved to aws_security_group.ec2_security_group
    resource "aws_security_group" "ec2_security_group" {
        id                     = "sg-0efae5be0072c3507"
        name                   = "myorg-dev-ec2-sg"
        tags                   = {}
        # (10 unchanged attributes hidden)
    }

Plan: 0 to add, 0 to change, 0 to destroy.

```

* Object has moved
* No destroy
* No recreate

Then run:

```
terraform apply
```

Terraform updates state internally.

Infrastructure remains untouched.

---

### After Successful Migration

Once applied successfully, the moved block can be removed.

It is only required during refactor.

---

# 7. Moving Resources Into Modules

Before:

```
aws_security_group.dev_sg
```

After:

```
module.ec2.aws_security_group.ec2_security_group
```

Moved block must reflect full address:

```
moved {
  from = aws_security_group.dev_sg
  to   = module.ec2.aws_security_group.ec2_security_group
}
```

Always verify exact address using:

```
terraform state list
```

---

# 8. Switching from count to for_each

Before:

```
aws_instance.app[0]
```

After:

```
aws_instance.app["app1"]
```

These are different addresses.

Each instance requires state migration.

Failure to handle this properly can recreate instances.

---

# 9. When to Use terraform state rm

If infrastructure should remain in AWS but no longer be managed by Terraform:

```
terraform state rm aws_security_group.ec2_security_group
```

Removes from state only.

Does not delete infrastructure.

---

# 10. When to Use terraform import

For practise purpose, I am removing the state manually

```
$ terraform state list
aws_instance.example["service-a-0"]
aws_instance.example["service-b-0"]
aws_instance.example["service-b-1"]
aws_security_group.ec2_security_group


$ terraform state rm aws_security_group.ec2_security_group
Acquiring state lock. This may take a few moments...
Removed aws_security_group.ec2_security_group
Successfully removed 1 resource instance(s).
Releasing state lock. This may take a few moments...

```

If infrastructure exists in AWS but not in state:

```
$ terraform import -var-file=env/dev.tfvars aws_security_group.ec2_security_group sg-0efae5be0072c3507

aws_security_group.ec2_security_group: Importing from ID "sg-0efae5be0072c3507"...
aws_security_group.ec2_security_group: Import prepared!
  Prepared aws_security_group for import
aws_security_group.ec2_security_group: Refreshing state... [id=sg-0efae5be0072c3507]

Import successful!

The resources that were imported are shown above. These resources are now in
your Terraform state and will henceforth be managed by Terraform.

Releasing state lock. This may take a few moments...

```

This brings existing infrastructure under Terraform management.


---

# 11. What Happens If You Ignore State Refactoring

Terraform may:

* Destroy resources
* Recreate security groups
* Replace EC2 instances
* Replace load balancers
* Replace databases
* Cause downtime
* Cause data loss

Understanding state prevents outages.

---

# 12. Safe Refactoring Checklist

Before renaming or restructuring:

1. Run terraform state list
2. Identify current addresses
3. Decide new addresses
4. Use moved block or terraform state mv
5. Run terraform plan
6. Confirm no unintended destroy
7. Apply carefully

---

# 13. Importance of Remote State Versioning

If using S3 backend with versioning enabled:

* You can restore previous state versions
* You can recover from mistakes
* You can inspect history

Backend versioning is critical for production safety.

---

# 14. Final Key Understanding

Terraform does NOT rename infrastructure.
Terraform renames state addresses.

Infrastructure identity in Terraform equals its resource address.

If the address changes, Terraform assumes the resource changed.

Safe refactoring requires explicitly telling Terraform how addresses moved.

Mastering this concept is essential for production-grade Terraform engineering.
