# Terraform Backend Bootstrap – Deep Explanation (Production Multi-Account Setup)

This document explains in detail what each Terraform file is doing and why it is structured this way.

This setup creates:

* S3 bucket (remote state storage)
* S3 versioning (state recovery protection)
* S3 encryption (security)
* S3 public access block (mandatory security)
* DynamoDB table (state locking)

This is executed once per AWS account.

---

# 📁 Project Structure

```
terraform-backend-bootstrap/
  ├── versions.tf
  ├── variables.tf
  ├── locals.tf
  ├── main.tf
  ├── dev.tfvars
  ├── test.tfvars
  └── prod.tfvars
```

Each file has a specific responsibility.

---

# 1️⃣ versions.tf – Tooling Control

```hcl
terraform {
  required_version = ">= 1.3.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 5.0"
    }
  }
}
```

What this does:

- Ensures Terraform CLI version compatibility
- Locks AWS provider version
- Prevents unexpected provider breaking changes

Why it matters:

Without version constraints, future provider updates may break infrastructure.

---

# 2️⃣ variables.tf – External Inputs

```hcl
variable "aws_region" {
  description = "AWS region where backend resources will be created"
  type        = string
}

variable "environment" {
  description = "Environment name (dev, test, prod, etc.)"
  type        = string
}

variable "project_name" {
  description = "Logical project or organization name"
  type        = string
}

variable "bucket_name_override" {
  description = "Optional custom bucket name. If null, a generated name will be used."
  type        = string
  default     = null
}

# NOTE:
# lifecycle.prevent_destroy cannot use variables because
# it must be known during configuration loading.
# So we DO NOT parameterize prevent_destroy.
# It will be hardcoded to true inside resources for safety.
```

What this file does:

- Defines what can change per account
- Keeps code reusable
- Removes hardcoding
- Makes environment switching clean

These values are provided through tfvars files.

---

# 3️⃣ locals.tf – Intelligent Naming Engine

```hcl
# Get AWS Account ID (used for global uniqueness)
data "aws_caller_identity" "current" {}

# Random suffix to guarantee globally unique S3 bucket name
resource "random_id" "bucket_suffix" {
  byte_length = 4
}

locals {
  # Standard naming prefix
  name_prefix = "${var.project_name}-${var.environment}"

  # S3 bucket must be globally unique across ALL AWS accounts
  bucket_name = var.bucket_name_override != null ? var.bucket_name_override : "${local.name_prefix}-tfstate-${data.aws_caller_identity.current.account_id}-${random_id.bucket_suffix.hex}"

  # DynamoDB table name (only needs account-level uniqueness)
  dynamodb_table_name = "${local.name_prefix}-tflock"
}
```

This file is extremely important.

It does not create infrastructure directly.
It calculates values used by other resources.

Detailed breakdown:

1️⃣ data "aws_caller_identity"

- Fetches AWS account ID dynamically
- Ensures bucket naming uniqueness per account

2️⃣ resource "random_id"

- Generates random hex suffix
- Guarantees global uniqueness
- Prevents S3 bucket name collision across AWS

3️⃣ locals block

- name_prefix → central naming logic
- bucket_name → auto-generates unique S3 bucket name
- dynamodb_table_name → deterministic lock table name

Why locals are powerful:

- Centralized naming
- No duplication of environment strings
- Single source of truth
- Clean, predictable resource names

# 4️⃣ main.tf – Infrastructure Creation

```hcl
provider "aws" {
  region = var.aws_region
}
```

Provider block:

- Connects Terraform to AWS
- Region is passed from variables
- Must appear only once in project

---

## S3 Bucket Resource

```hcl
resource "aws_s3_bucket" "terraform_state" {
  bucket = local.bucket_name

  tags = {
    Name        = local.bucket_name
    Environment = var.environment
    Project     = var.project_name
  }

  lifecycle {
    prevent_destroy = true
  }
}
```

What this does:

- Creates remote state bucket
- Uses computed unique name
- Applies structured tags
- Enables prevent_destroy

prevent_destroy:

- Stops accidental deletion
- Protects entire infrastructure state

---

## Versioning Block

```hcl
resource "aws_s3_bucket_versioning" "versioning" {
  bucket = aws_s3_bucket.terraform_state.id

  versioning_configuration {
    status = "Enabled"
  }
}
```

Why versioning matters:

- Every state update creates a new version
- Enables rollback of corrupted state
- Critical for production safety

---

## Encryption Block

```hcl
resource "aws_s3_bucket_server_side_encryption_configuration" "encryption" {
  bucket = aws_s3_bucket.terraform_state.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}
```

What this does:

- Encrypts state file at rest
- Protects sensitive infrastructure metadata

---

## Public Access Block

```hcl
resource "aws_s3_bucket_public_access_block" "block_public" {
  bucket = aws_s3_bucket.terraform_state.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}
```

This ensures:

- State file cannot be exposed publicly
- Prevents security misconfiguration

---

## DynamoDB Lock Table

```hcl
resource "aws_dynamodb_table" "terraform_lock" {
  name         = local.dynamodb_table_name
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }

  tags = {
    Name        = local.dynamodb_table_name
    Environment = var.environment
    Project     = var.project_name
  }

  lifecycle {
    prevent_destroy = true
  }
}
```

Purpose:

- Prevents concurrent terraform apply operations
- Avoids state corruption
- Ensures only one process updates state at a time

# 🗑 How To Destroy The Backend (If Ever Needed)

Normally, backend infrastructure is NOT destroyed.
It is meant to live permanently in the AWS account.

However, if you really need to remove it (for example, closing an account or cleaning up a lab environment), follow these steps.

---

## Why You Cannot Destroy Immediately

Both S3 bucket and DynamoDB table have:

```
lifecycle {
  prevent_destroy = true
}
```

This blocks accidental deletion.
If you run:

```
terraform destroy
```

Terraform will fail intentionally.

---

## Step‑by‑Step Safe Destroy Process

### 1️⃣ Disable Protection

Edit `main.tf` and change:

```
lifecycle {
  prevent_destroy = true
}
```

To:

```
lifecycle {
  prevent_destroy = false
}
```

Do this for:

* aws_s3_bucket
* aws_dynamodb_table

---

### 2️⃣ Apply The Change

```
terraform apply --var-file=dev.tfvars
```

This updates the lifecycle rule in state.

---

### 3️⃣ Destroy Resources

```
terraform destroy --var-file=dev.tfvars
```

---

## Important About Versioned Buckets

Because versioning is enabled:

- If state files exist in the bucket, Terraform may not delete it.
- All object versions must be removed first.

If destroy fails:

* Go to S3 console
* Delete all object versions
* Then run destroy again

---

# When Should Backend Be Destroyed?

- Closing AWS account
- Cleaning up temporary lab environment
- Rebuilding entire infrastructure from scratch

In normal production environments, backend infrastructure remains permanent.

---

#  Why Changing prevent_destroy Shows "No Changes"

When you modify this block in `main.tf`:

```
lifecycle {
  prevent_destroy = true
}
```

and change it to:

```
lifecycle {
  prevent_destroy = false
}
```

Then run:

```
terraform apply --var-file=dev.tfvars
```

Terraform may show:

```
No changes. Your infrastructure matches the configuration.
```

This is expected behavior.

Reason:

- `prevent_destroy` is NOT an AWS setting.
- It is a Terraform behavior rule.
- It does not modify S3 or DynamoDB.
- It only changes how Terraform allows or blocks destroy operations.

Terraform compares:

Configuration  vs  Real AWS Infrastructure

Since lifecycle rules do not exist in AWS, there is nothing to update in the cloud.

Therefore:

- No infrastructure change is required
- No update is applied
- You can immediately run `terraform destroy`

This is normal and confirms Terraform is working correctly.
