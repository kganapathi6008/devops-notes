# Module 3 – State Management & Meta Arguments

This module explains how Terraform tracks infrastructure using state and how to control resource behavior using meta-arguments.

---

# PART 1 – TERRAFORM STATE

## 1. What is terraform.tfstate?

Terraform state is a JSON file that stores:

* What resources Terraform created
* Their real-world IDs
* Their current attributes
* Dependency relationships

Example:

If you create an EC2 instance:

```hcl
resource "aws_instance" "app_server" {
  ami           = "ami-123456"
  instance_type = "t3.micro"
}
```

Terraform stores in state:

* Instance ID (i-0abcd1234)
* Public IP
* Subnet ID
* Security group

Without state, Terraform would not know what it already created.

---

## 2. Local State vs Remote State

### Local State (Default)

Stored as:

```
terraform.tfstate
```

Problems:

* Not safe for teams
* Can be deleted accidentally
* No locking (two people can apply at same time)

---

### Remote State

State stored in remote storage like:

* S3
* Azure Storage
* GCS

Benefits:

* Shared across team
* State locking
* Safer
* Versioning support

---

# PART 2 – BACKEND CONFIGURATION

Backend tells Terraform **where to store state**.

Example: S3 Backend

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock-table"
    encrypt        = true
  }
}
```

Explanation:

* bucket → S3 bucket name
* key → path inside bucket
* region → AWS region
* dynamodb_table → for locking
* encrypt → enable encryption

After adding backend:

```
terraform init
```

Terraform migrates local state to S3.

# PART 3 – DYNAMODB STATE LOCKING

Why locking?

If two engineers run `terraform apply` at same time:

* State can get corrupted

Solution: DynamoDB locking

Create table:

```hcl
resource "aws_dynamodb_table" "terraform_lock" {
  name         = "terraform-lock-table"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }
}
```

Now only one person can modify state at a time.

# PART 4 – TERRAFORM STATE COMMANDS

## terraform state list

Lists all resources in state.

```
terraform state list
```

Example output:

```
aws_instance.app_server
aws_security_group.app_sg
```

---

## terraform state show

Shows detailed information about a resource.

```
terraform state show aws_instance.app_server
```

---

## terraform state rm

Removes resource from state WITHOUT deleting actual resource.

```
terraform state rm aws_instance.app_server
```

Use carefully.

# PART 5 – META ARGUMENTS

Meta arguments control how resources behave.

---

## 1. count

Creates multiple identical resources.

```hcl
resource "aws_instance" "app_server" {
  count         = 2
  ami           = "ami-123456"
  instance_type = "t3.micro"
}
```

Access individual instance:

```
aws_instance.app_server[0]
aws_instance.app_server[1]
```

Use when:

* Resources are identical

Limitation:

* Index-based (can cause recreation issues)

---

## 2. for_each

Creates multiple resources using map or set.

Example using map:

```hcl
variable "instances" {
  default = {
    app1 = "t3.micro"
    app2 = "t3.small"
  }
}

resource "aws_instance" "app_server" {
  for_each      = var.instances
  ami           = "ami-123456"
  instance_type = each.value

  tags = {
    Name = each.key
  }
}
```

Better than count because:

* Uses keys
* No index shifting problem

---

## 3. depends_on

Explicit dependency.

```hcl
resource "aws_instance" "app" {
  ami           = "ami-123456"
  instance_type = "t3.micro"

  depends_on = [aws_security_group.app_sg]
}
```

Use only when implicit dependency is not detected.

---

## 4. lifecycle

Controls creation and destruction behavior.

### create_before_destroy

```hcl
lifecycle {
  create_before_destroy = true
}
```

Creates new resource before destroying old one.

---

### prevent_destroy

```hcl
lifecycle {
  prevent_destroy = true
}
```

Prevents accidental deletion.

---

### ignore_changes

```hcl
lifecycle {
  ignore_changes = [tags]
}
```

Terraform ignores changes to tags.

---

# PART 6 – DYNAMIC BLOCKS

Used when nested blocks must be generated dynamically.

Example: Multiple ingress rules

```hcl
variable "ports" {
  default = [22, 80, 443]
}

resource "aws_security_group" "app_sg" {
  name = "app-sg"

  dynamic "ingress" {
    for_each = var.ports
    content {
      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}
```

Without dynamic block, you would manually write multiple ingress blocks.

1. Create S3 bucket
2. Create DynamoDB table
3. Configure backend block
4. Run terraform init
5. Use for_each to create multiple instances
6. Add lifecycle rules
7. Verify state using terraform state list
