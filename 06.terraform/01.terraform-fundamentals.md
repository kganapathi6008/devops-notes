# 1. Infrastructure as Code (IaC)

## What is Infrastructure as Code

Infrastructure as Code (IaC) is the practice of provisioning and managing infrastructure using configuration files instead of manual processes.

Instead of:

* Logging into AWS Console
* Clicking buttons
* Manually creating servers

We define infrastructure using code:

```hcl
resource "RESOURCE_TYPE" "LOGICAL_NAME" {
  argument_name = argument_value
}
```

```hcl
resource "aws_instance" "example" {
  ami           = "ami-12345678"
  instance_type = "t3.micro"
}
```

Terraform reads this configuration and creates infrastructure automatically.

## Why IaC is Important

* Automation
* Version control using Git
* Reusability
* Consistency across environments
* Faster deployments
* Reduced human error
* Easy rollback using code history

# 2. Imperative vs Declarative

## Imperative Approach

In the imperative approach, you define how to create infrastructure step by step.

Example steps:

* Create VPC
* Create Subnet
* Create EC2
* Attach security group

You control the exact execution order.

Examples of imperative tools:

* Shell scripts
* AWS CLI scripts

## Declarative Approach

In the declarative approach, you define what you want.

Example:
"I need an EC2 inside a VPC"

Terraform determines:

* The correct creation order
* Resource dependencies
* Execution plan

Terraform is declarative.

# 4. Terraform Architecture

Terraform follows a Core and Provider plugin architecture.

## High Level Flow

1. You write Terraform configuration files (.tf)
2. Terraform Core reads the configuration
3. Core builds a dependency graph
4. Core generates an execution plan
5. Providers interact with cloud APIs
6. Infrastructure is created or updated
7. State file is updated

## Terraform Core

Terraform Core is the main engine.

Responsibilities:

* Parse HCL configuration files
* Build dependency graph
* Generate execution plan
* Maintain state
* Communicate with providers via RPC

Terraform Core does not directly communicate with AWS or Azure. It communicates through providers.

## Providers

Providers are plugins that interact with external APIs.

Examples:

* AWS provider interacts with AWS APIs
* Azure provider interacts with Azure APIs
* Kubernetes provider interacts with Kubernetes API server

Example:

```hcl
provider "aws" {
  region  = "us-east-1"
  profile = "default"
}
```

Providers are downloaded during:

```
terraform init
```

They are stored inside the .terraform directory.

## State

Terraform stores infrastructure mapping inside:

```
terraform.tfstate
```

State contains:

* Resource IDs
* Resource attributes
* Metadata
* Dependency relationships

Without state, Terraform cannot track existing infrastructure.

# 6. HCL Basics

Terraform uses HCL (HashiCorp Configuration Language).

HCL is:

* Human readable
* Declarative
* Structured

## Blocks

Blocks define infrastructure components.

General format:

```hcl
block_type "label1" "label2" {
  argument = value
}
```

Common block types:

* resource
* provider
* variable
* output
* module

Example:

```hcl
resource "aws_instance" "web" {
  instance_type = "t3.micro"
}
```

## Arguments

Arguments are key-value pairs inside blocks.

```hcl
instance_type = "t3.micro"
```

Left side is argument name. Right side is value.

## Data Types

Terraform supports:

* string: "hello"
* number: 10
* bool: true
* list: ["a", "b"]
* map: { key = "value" }
* object

Example:

```hcl
tags = {
  Name = "web-server"
  Env  = "dev"
}
```

## Expressions

Expressions allow dynamic values instead of hardcoded values.

### a. Reference Expression

Used to refer to another resource or variable.

```hcl
vpc_security_group_ids = [aws_security_group.sg.id]
```

Another example:

```hcl
subnet_id = aws_subnet.public.id
```

### b. Conditional Expression

Used to apply logic based on a condition.

```hcl
instance_type = var.environment == "prod" ? "t3.medium" : "t3.micro"
```

If environment equals "prod", Terraform uses t3.medium. Otherwise, it uses t3.micro.


### Built-in Function Expression

Terraform provides many built-in functions.

```hcl
name = upper("terraform")
```

This converts "terraform" into "TERRAFORM".

More examples:

```hcl
length(var.instance_list)
```

Returns the number of items in a list.

```hcl
lower("DEV")
```

Returns "dev".

### c. Arithmetic Expression

Used for numeric calculations.

```hcl
count = var.instance_count + 1
```

### d. Logical Expression

Used with boolean values.

```hcl
enabled = var.is_production && var.create_resource
```

### e. For Expression

Used to loop over collections.

```hcl
[for name in var.user_names : upper(name)]
```

If input is ["dev", "test"], output becomes ["DEV", "TEST"].

## Interpolation

Terraform automatically evaluates expressions.

Old style:

```
"${aws_instance.web.id}"
```

Modern style:

```
aws_instance.web.id
```

## Comments

Single line comments:

```hcl
# This is a comment
// This is also a comment
```

Multi-line comments:

```hcl
/*
Multi-line comment
*/
```

## Multi-line Strings (HEREDOC)

Used for scripts such as user_data.

```hcl
user_data = <<EOF
#!/bin/bash
echo "Hello"
EOF
```

# 7. Provider and Resource Blocks

## Provider Block

Defines which platform Terraform connects to.

```hcl
provider "aws" {
  region = "us-east-1"
}
```

## Resource Block

Defines an infrastructure object.

Structure:

```hcl
resource "provider_resource" "name" {
  argument = value
}
```

Example:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.micro"
}
```

aws_instance is the resource type.
web is the local name.

# 8. Implicit Dependencies

Terraform automatically understands dependencies when one resource references another.

Example:

```hcl
resource "aws_security_group" "sg" {
  name = "web-sg"
}

resource "aws_instance" "web" {
  ami           = "ami-123456"
  instance_type = "t3.micro"
  vpc_security_group_ids = [aws_security_group.sg.id]
}
```

EC2 depends on the security group. Terraform creates the security group first automatically.

This is called implicit dependency.

Explicit dependency can be defined using:

```hcl
depends_on = [aws_security_group.sg]
```

# Terraform Project Structure Example

This is a clean and professional Terraform project layout using separate files.

Project Structure:

* output.tf
* provider.tf
* versions.tf
* network.tf
* compute.tf

All `.tf` files in the same directory are automatically loaded and treated as a single configuration.

---

## versions.tf

Defines Terraform and provider version constraints.

```hcl
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }
}
```

---

## provider.tf

Defines provider configuration.

```hcl
provider "aws" {
  region = "us-east-1"
}
```

---

## network.tf

Defines networking resources such as Security Groups.

```hcl
resource "aws_security_group" "ssh_sg" {
  name        = "ssh-sg"
  description = "Allow SSH access"

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "ssh-security-group"
  }
}
```

---

## compute.tf

Defines compute resources such as EC2 instances.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0f3caa1cf4417e51b"
  instance_type = "t3.micro"

  vpc_security_group_ids = [aws_security_group.ssh_sg.id]

  tags = {
    Name = "Terraform-Server"
  }
}
```

Note:
Replace the AMI ID with a valid AMI available in your selected region.

---

## output.tf

The main.tf file can be empty or used for high-level orchestration. Or you can keep `output.tf` file.

Example (optional output block):

```hcl
output "instance_id" {
  value = aws_instance.web.id
}
```

---

## How Terraform Treats These Files

Terraform automatically:

* Reads all `.tf` files in the directory
* Merges them into a single configuration
* Builds one dependency graph
* Applies them together


This structure follows real-world production best practices and keeps your code organized and scalable.

# **Clear explanation of commands**
# Step 4: Initialize Terraform

Run the following command inside your project directory:

```
terraform init
```

## What Happened When You Ran terraform init

When you executed the command, Terraform performed several internal steps.

### a. It Checked Your Configuration

You defined the AWS provider as:

```hcl
provider "aws" {
  region = "us-east-1"
}
```

You did not specify a provider version.

---

### b. Terraform Looked for the AWS Provider

Terraform automatically searched the Terraform Registry for:

```
hashicorp/aws
```

Since no version constraint was defined, Terraform selected the latest available version.

From your output:

```
Finding latest version of hashicorp/aws...
Installing hashicorp/aws v6.34.0...
```

Terraform selected version v6.34.0 and downloaded it.

---

### c. It Downloaded the Provider Plugin

The AWS provider plugin was downloaded into the:

```
.terraform/
```

This directory contains:

* Provider binaries
* Plugin metadata

This folder should not be committed to Git.

---

### d. It Created .terraform.lock.hcl

Terraform generated a lock file:

```
.terraform.lock.hcl
```

From your output:

```
Terraform has created a lock file .terraform.lock.hcl to record the provider selections it made above.
```

This file:

* Locks the provider version (v6.34.0)
* Ensures consistency across environments
* Prevents automatic upgrades
* Guarantees the same provider version for all team members

Even though you did not specify a version, Terraform:

* Selected the latest version
* Locked it
* Recorded it in the lock file

---

## Why This Is Important

If a new major version of the AWS provider is released in the future, Terraform will not automatically upgrade and break your configuration. The lock file protects your project from unexpected changes.

---

## Production Best Practice

In real projects, it is recommended to define a version constraint explicitly:

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }
}
```

This ensures:

* Only version 6.x is used
* No automatic upgrade to 7.x

Terraform will then download the latest compatible 6.x version and lock it in .terraform.lock.hcl.

---

## How to Upgrade Provider Version Properly

If a new provider version is released and you want to upgrade safely, follow this process.

### Step 1: Update Version Constraint

Modify your configuration to allow the new version. Create `versions.tf` file

Example: upgrading from 6.x to 7.x

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 7.0"
    }
  }
}
```

---

### Step 2: Run Upgrade Command

Run:

```
terraform init -upgrade
```

This command:

* Checks for newer allowed versions
* Downloads updated provider
* Updates .terraform.lock.hcl

---

### Step 3: Review Changes

Run:

```
terraform plan
```

Carefully review:

* Any deprecation warnings
* Any breaking changes
* Resource behavior changes

---

### Step 4: Test in Lower Environment

Always test upgrades in:

* Development environment
* Staging environment

Never upgrade directly in production.

---

### Important Notes About Older Versions

* Older provider versions usually remain available in Terraform Registry.
* Even if a new major version is released, your locked version will continue working.
* If a version is deprecated, Terraform may show warnings but will still work.
* If a version becomes unavailable (rare case), terraform init will fail and you must upgrade.

# Step 6: Preview Infrastructure

Run the following command inside your project directory:

```
terraform plan
```

This command does NOT create any resources.

# What terraform plan Does Internally

When you run `terraform plan`, Terraform performs the following steps:

1. Reads all `.tf` configuration files
2. Reads the current state file (terraform.tfstate)
3. Compares configuration with real infrastructure
4. Generates an execution plan

---

# Understanding Your Output

At the top, Terraform shows:

```
Terraform used the selected providers to generate the following execution plan.
Resource actions are indicated with the following symbols:
  + create
```

Symbol meanings:

* `+` means resource will be created
* `~` means resource will be modified
* `-` means resource will be destroyed

---

Terraform then displayed:

```
Plan: 2 to add, 0 to change, 0 to destroy.
```

This means:

* 2 resources will be created
* 0 resources will be modified
* 0 resources will be deleted

In your case:

* aws_security_group.ssh_sg will be created
* aws_instance.web will be created

---

# What Does "known after apply" Mean

You will see many fields like:

```
(known after apply)
```

This means:

* Terraform does not know this value yet
* AWS will generate it during creation
* After running `terraform apply`, these values will be stored in the state file

Examples:

* id
* public_ip
* arn
* availability_zone

---

# Important Note About -out Option

At the bottom, Terraform showed:

```
Note: You didn't use the -out option to save this plan
```

If you want to ensure the exact same plan is applied, you should run:

```
terraform plan -out=tfplan
```

Then apply using:

```
terraform apply tfplan
```

This guarantees that the exact reviewed plan is executed.

---

# Why Reviewing Plan is Important

Always review:

* Resources being created
* Any unexpected changes
* Any resources being destroyed

In production environments, running `terraform plan` before `terraform apply` is mandatory.

---

# Summary

terraform plan:

* Does not create resources
* Shows proposed infrastructure changes
* Displays create, modify, destroy actions
* Helps avoid mistakes before applying changes

# Files and Folders Generated During init, plan, and apply

When you run the following commands:

* terraform init
* terraform plan
* terraform apply

Terraform generates several important files and folders.

---

## 1. .terraform/ Directory

Created during:

```
terraform init
```

Purpose:

* Stores downloaded provider plugins
* Stores module cache (if modules are used)
* Contains provider binaries

Example structure:

```
.terraform/
  providers/
    registry.terraform.io/
```

Important:

* This folder can be safely deleted
* It will be recreated on next terraform init
* It should NOT be committed to Git

---

## 2. .terraform.lock.hcl

Created during:

```
terraform init
```

Purpose:

* Locks the exact provider version (for example 6.34.0)
* Ensures consistency across environments
* Prevents accidental upgrades

This file should be committed to version control.

---

## 3. terraform.tfstate

Created during:

```
terraform apply
```

This is the most important file.

It stores:

* Resource IDs
* Public IP address
* ARN values
* Current configuration snapshot
* Dependency information

Terraform compares:

* Configuration files
* terraform.tfstate
* Real infrastructure

to determine future changes.

Important:

* Contains sensitive information
* Should NOT be committed if using local state
* In production, use remote backend (example: S3)

---

## 4. terraform.tfstate.backup

Created automatically when state changes.

Purpose:

* Backup of previous state file
* Used for recovery if state becomes corrupted

---

## 5. Saved Plan File (Optional)

If you run:

```
terraform plan -out=tfplan
```

Terraform creates a file named tfplan.

Purpose:

* Stores the exact execution plan
* Can be applied using:

```
terraform apply tfplan
```

This guarantees that the reviewed plan is exactly what gets executed.

This file should not be committed to Git.
