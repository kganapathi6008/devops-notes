# Managing Multiple Backends and Multiple tfvars Using Separate Folders (Single ec2-project)

This structure keeps backend configuration files and tfvars files inside their own dedicated folders for better organization and clarity.

---

# Target Folder Structure

```
ec2-project/
  ├── backend/
  │     ├── backend-dev.hcl
  │     ├── backend-test.hcl
  │     └── backend-prod.hcl
  │
  ├── env/
  │     ├── dev.tfvars
  │     ├── test.tfvars
  │     └── prod.tfvars
  │
  ├── backend.tf
  ├── versions.tf
  ├── main.tf
  └── variables.tf
```

This keeps:

* Backend configuration separated
* Environment variables separated
* Project code clean at root level

---

# backend.tf (Common Declaration)

```hcl
terraform {
  backend "s3" {}
}
```

This only declares that S3 backend is used.
Actual backend values are loaded from backend/*.hcl files during initialization.

# versions.tf

```hcl

  required_version = ">= 1.3.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 5.0"
    }
  }

```

```hcl
  type = string

```

# main.tf

```hcl
provider "aws" {
  region = var.aws_region
}

resource "aws_instance" "example" {
  ami           = "ami-0f3caa1cf4417e51b"
  instance_type = var.instance_type

  tags = {
    Name        = "ec2-${var.environment}-instance"
    Environment = var.environment
  }
}
```

# Step 6: Initialize Per Environment

Because backend block is empty, we pass backend details during init.

Example for dev:

```
terraform init -reconfigure \
  -backend-config="bucket=myorg-dev-tfstate-703176829635-00631489" \
  -backend-config="key=ec2/dev/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=myorg-dev-tflock" \
  -backend-config="encrypt=true"
```

For test:

```
terraform init -reconfigure \
  -backend-config="bucket=myorg-dev-tfstate-703176829635-00631489" \
  -backend-config="key=ec2/test/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=myorg-dev-tflock" \
  -backend-config="encrypt=true"
```

For prod:

```
terraform init -reconfigure \
  -backend-config="bucket=myorg-dev-tfstate-703176829635-00631489" \
  -backend-config="key=ec2/prod/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=myorg-dev-tflock" \
  -backend-config="encrypt=true"
```

Each environment gets a separate state file because the `key` changes.

---

# How to Initialize Per Environment using HCL files

For dev:

```
terraform init -reconfigure -backend-config=backend/backend-dev.hcl
```

For test:

```
terraform init -reconfigure -backend-config=backend/backend-test.hcl
```

For prod:

```
terraform init -reconfigure -backend-config=backend/backend-prod.hcl
```

You must re-run `terraform init -reconfigure` when switching environments. (Even when doing the `terraform apply` or `terraform destroy` )



---

# How to Apply Per Environment

For dev:

```
terraform apply --var-file=env/dev.tfvars --auto-approve
```

For test:

```
terraform apply --var-file=env/test.tfvars --auto-approve
```

For prod:

```
terraform apply --var-file=env/prod.tfvars --auto-approve
```

# What Happens Internally

* Same project code
* Same backend infrastructure
* Different state keys
* Different tfvars
* Fully isolated environments

Inside S3 bucket you will see:

```
ec2/
  dev/terraform.tfstate
  test/terraform.tfstate
  prod/terraform.tfstate
```

Each environment maintains its own state file.

---

# Benefits of This Structure

* Clean separation of concerns
* Backend configs grouped together
* Environment variables grouped together
* Root folder contains only core Terraform code
* Easy for teams to understand
* Scales well as environments grow

---

# Final Pattern

Inside one ec2-project folder:

* backend/ contains all backend configuration files
* env/ contains all tfvars files
* Root contains Terraform code
* Unique state key per environment
* Reinitialize when switching environments

This is a structured and scalable way to manage multiple environments in a single Terraform project.
