# Terraform State Concepts and EC2 Example

You have already created:

* S3 bucket for remote state
* DynamoDB table for state locking

This document explains the core Terraform state concepts clearly and then shows how to create an EC2 instance using your remote backend.

---

# What is terraform.tfstate?

`terraform.tfstate` is a JSON file that Terraform uses to track infrastructure.

It stores:

* Resources Terraform created
* Real AWS resource IDs
* Current attribute values
* Dependency relationships

Example:

If Terraform creates an EC2 instance, the state file stores:

* Instance ID (i-xxxxxxxx)
* Public IP address
* Subnet ID
* Security group ID
* Tags

Why state is required:

Terraform does not query AWS to understand what it originally created. Instead, it compares:

Configuration  vs  State file

Without the state file, Terraform would not know what it manages.

---

# Local State vs Remote State

## Local State

Default behavior.

State file stored as:

```
terraform.tfstate
```

Problems:

* Not safe for teams
* Can be deleted accidentally
* No locking mechanism
* Difficult collaboration

---

## Remote State

State stored in remote storage like:

* S3
* Azure Storage
* GCS

Benefits:

* Shared across team members
* Safe collaboration
* Centralized state management
* Locking support
* Versioning support

In your case:

* S3 stores state
* DynamoDB provides locking

---

# Backend Configuration

Backend tells Terraform where to store the state file.

Example backend configuration using your created infrastructure:

```hcl
terraform {
  backend "s3" {
    bucket         = "myorg-dev-tfstate-703176829635-00631489"
    key            = "ec2-project/dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "myorg-dev-tflock"
    encrypt        = true
  }
}
```

Explanation:

* bucket: S3 bucket name
* key: Logical path inside bucket
* region: AWS region
* dynamodb_table: Lock table
* encrypt: Encrypt state operations

Backend must be configured before running `terraform init`.

---

# Why DynamoDB State Locking?

Problem:

If two engineers run `terraform apply` at the same time:

* Both read same state
* Both attempt updates
* State corruption can occur

Solution:

DynamoDB locking ensures:

* Only one apply runs at a time
* Other users must wait
* State integrity is preserved

During apply:

* Terraform creates lock entry in DynamoDB
* Performs operations
* Removes lock after completion

# Terraform State Commands

## terraform state list

Lists all resources tracked in state.

```
terraform state list
```

---

## terraform state show

Shows detailed state of a specific resource.

```
terraform state show aws_instance.example
```

---

## terraform state rm

Removes resource from state without deleting real infrastructure.

```
terraform state rm aws_instance.example
```

Used when importing or detaching resources.

---

## terraform import

Imports existing AWS resource into Terraform state.

```
terraform import aws_instance.example i-1234567890abcdef0
```

# backend.tf

```hcl
terraform {
  backend "s3" {
    bucket         = "myorg-dev-tfstate-703176829635-00631489"
```

# versions.tf

```hcl

  required_version = ">= 1.3.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 5.0"
    }
  }

```

---

# variables.tf

```hcl

  type = string

  type = string
```

```hcl

  region = var.aws_region

    Name = "ec2-demo-instance"
```

# Steps to Deploy

Inside ec2-project folder:

```
terraform init
terraform apply --var-file=dev.tfvars
```

What happens:

* Backend connects to S3
* Lock is created in DynamoDB
* EC2 instance is created
* State is stored in S3
* Lock is released

# IMPORTANT: Can We Use Variables Inside Backend Block?

Very common doubt:

Can we write like this?

```hcl
terraform {
  backend "s3" {
    bucket = var.bucket_name   # ❌ This will NOT work
  }
}
```

Answer: ❌ NO.

Why?

Terraform works in two phases:

1. `terraform init` → Backend initializes first
2. `terraform plan/apply` → Variables are loaded later

Backend runs BEFORE variables are evaluated.

So Terraform does not allow `var.*` inside backend block.

---

# How To Handle This Correctly (Simple & Practical)

Instead of using variables, we use one of the following real-world methods.

---

## Method 1 – Partial Backend + CLI Arguments (Very Common)

Keep backend block empty:

```hcl
terraform {
  backend "s3" {}
}
```

Then initialize per environment:

DEV:

```
terraform init \
  -backend-config="bucket=my-terraform-remote-state-bucket" \
  -backend-config="key=dev/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=terraform-lock-table" \
  -backend-config="encrypt=true"
```

TEST:

```
terraform init \
  -backend-config="bucket=my-terraform-remote-state-bucket" \
  -backend-config="key=test/terraform.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=terraform-lock-table" \
  -backend-config="encrypt=true"
```

Notice:
Only the `key` changes per environment.

---

## Method 2 – Separate backend.hcl Files (Cleaner Approach)

Create files:

```
backend-dev.hcl
backend-test.hcl
backend-prod.hcl
```

Example: backend-dev.hcl

```
bucket         = "my-terraform-remote-state-bucket"
key            = "dev/terraform.tfstate"
region         = "us-east-1"
dynamodb_table = "terraform-lock-table"
encrypt        = true
```

Initialize:

```
terraform init -backend-config=backend-dev.hcl
```

For prod:

```
terraform init -backend-config=backend-prod.hcl
```

This approach is:

- Cleaner
- Easier for CI/CD
- Safer for production

---

# Final Understanding

Backend configuration:

* Cannot use Terraform variables
* Must be decided during `terraform init`
* Should be separated per environment using key or backend files

This is how real-world multi-environment Terraform projects manage remote state safely.
