# Helm
## 1. What is Helm?

Helm is a **package manager for Kubernetes**. It helps us install, upgrade, downgrade (rollback), and manage applications running on Kubernetes in an easy and organized way.

In simple words:

* Kubernetes uses **YAML files** (Deployment, Service, HPA, ConfigMap, etc.)
* Writing and maintaining many YAML files manually is hard
* **Helm solves this problem** by packaging all those YAMLs together and making them configurable

Think of Helm like:

* `apt` or `yum` for Linux
* `npm` for Node.js
* `pip` for Python

For Kubernetes → **Helm**

---

## 2. Why Do We Need Helm?

### Problems without Helm

Without Helm:

* We need to maintain **many YAML files**
* Same YAML copied for dev, qa, prod (duplication)
* Updating image versions is manual and error-prone
* Rollback to previous version is difficult
* No easy way to track what version is deployed

### Benefits of Helm

With Helm:

* One reusable **chart** for all environments
* Values can be changed using `values.yaml` or `--set`
* Easy **upgrade and rollback**
* Helm keeps **release history**
* Very useful in **CI/CD pipelines**

---

## 3. Does Helm Maintain Versioning of Our Application?

✅ **Yes, Helm maintains versioning**.

Helm tracks:

* Chart version (`Chart.yaml`)
* Application version (`appVersion`)
* Release revisions (1, 2, 3, ...)

Every time you run:

```bash
helm upgrade
```

Helm:

* Creates a **new revision**
* Stores previous versions
* Allows rollback to any previous revision

You can check history using:

```bash
helm history <release-name>
```

---

## 4. Helm Versions (Helm 2 vs Helm 3)

### Helm 3 (Current & Recommended)

Helm is a client-only tool (Helm v3+).

* **No Tiller**
* Helm CLI (local machine / CI)
* Directly talks to Kubernetes API
* Uses Kubernetes **Secrets** to store release data
* Better security

👉 Today, we **always use Helm 3**

### Helm 2 (Old – Not Used Now)

Helm 2 had two components:

* Helm Client (CLI)
* **Tiller** (server-side component running inside cluster)

Problems with Helm 2:

* Security risk (Tiller had cluster-wide access)
* Extra component to manage
* RBAC complexity


## 7. Core Helm Concepts

### 7.1 Chart

A **Chart** is a Helm package containing everything needed to run an app on Kubernetes. A **chart** is a collection of Kubernetes YAML files written as templates.


A chart includes:

* Kubernetes manifests (templated)
* Default configuration values
* Metadata about the application

### 7.2 Release

A **Release** is a running instance of a chart in a Kubernetes cluster.

* Same chart can be installed multiple times
* Same chart + different values = different releases

### 7.3 Repository

A **Helm Repository** is a collection of charts.

Examples:

* Public repos (Bitnami, Prometheus)
* Private internal repos

## 8. Creating a Custom Helm Chart

### Create Chart

```bash
helm create java-web-app-chart
```

This generates a standard chart structure.

### Directory Structure

```text
java-web-app-chart/
├── Chart.yaml        # Chart metadata
├── values.yaml       # Default values
├── charts/           # Dependencies
├── templates/        # Kubernetes manifests
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── hpa.yaml
│   ├── serviceaccount.yaml
│   └── _helpers.tpl
└── README.md
```

### Important Files

**Chart.yaml**

* name, version, appVersion

**values.yaml**

* Default configuration values

**templates/**

* Go-templated Kubernetes YAML
---

## 9. Understanding values.yaml

`values.yaml` contains **default configuration**.

Example:

```yaml
replicaCount: 1

image:
  repository: myrepo/java-app
  tag: 1

autoscaling:
  enabled: false

serviceAccount:
  create: true
```

---

## 10. Rendering Templates (Dry Run)

### View Manifests Without Deploying

```bash
helm template java-web-app-chart/
```

This command:

* Converts templates into plain Kubernetes YAML
* Does NOT deploy anything

---

### Enable HPA (Autoscaling)

```bash
helm template java-web-app-chart/ --set autoscaling.enabled=true
```

By default:

* HPA is **disabled**

---

### Disable ServiceAccount

```bash
helm template java-web-app-chart/ \
--set autoscaling.enabled=true \
--set serviceAccount.create=false
```

---

## 11. Installing the Application

### Basic Install

```bash
helm install java-web-app java-web-app-chart/
```

* `java-web-app` → release name
* `java-web-app-chart/` → chart path

---

### Install with Custom Values

```bash
helm install java-web-app java-web-app-chart/ \
--set image.tag=2 \
--set autoscaling.enabled=true \
--set serviceAccount.create=false \
--set replicaCount=5 \
-n default
-f values-dev.yaml
```

---

## 12. Upgrading the Application

```bash
helm upgrade java-web-app java-web-app-chart/ \
--set image.tag=3 \
--set autoscaling.enabled=true \
--set serviceAccount.create=false \
--set replicaCount=5 \
-n default
-f values-dev.yaml
```

What happens:

* New revision is created
* Only changed resources are updated
* Old version is kept for rollback

---

## 13. Rollback to Previous Version

### View History

```bash
helm history java-web-app
```

### Rollback

```bash
helm rollback java-web-app 1
```

This restores revision 1.

---

## 14. Uninstalling the Application

```bash
helm uninstall java-web-app -n default
```

* Deletes all Kubernetes resources
* Release history is removed

## 16. _helpers.tpl – Reusable Logic in Helm

### What is _helpers.tpl?

`_helpers.tpl` is a special Helm template file used to **store reusable template logic**. It helps you avoid repeating the same code in multiple YAML files.

In simple terms:

* It acts like a **function library** for Helm templates
* You define reusable snippets once
* You reuse them everywhere using `include` or `template`

Files starting with `_` are **not rendered directly** into Kubernetes manifests.

---

### Why Do We Need _helpers.tpl?

Without `_helpers.tpl`:

* Same labels repeated in Deployment, Service, HPA, etc.
* High chance of mistakes and inconsistency
* Hard to maintain when changes are needed

With `_helpers.tpl`:

* Single source of truth
* Cleaner templates
* Easier maintenance

---

### Common Use Cases

Typically used for:

* Application name
* Full resource name
* Standard labels
* Selector labels
* ServiceAccount name

---

### Example: Defining Helpers

**templates/_helpers.tpl**

```yaml
{{/* Chart name */}}
{{- define "java-web-app-chart.name" -}}
{{ .Chart.Name }}
{{- end }}

{{/* Full name */}}
{{- define "java-web-app-chart.fullname" -}}
{{ printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/* Common labels */}}
{{- define "java-web-app-chart.labels" -}}
app.kubernetes.io/name: {{ include "java-web-app-chart.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}
```

---

### Using Helpers in Templates

**deployment.yaml**

```yaml
metadata:
  name: {{ include "java-web-app-chart.fullname" . }}
  labels:
{{ include "java-web-app-chart.labels" . | indent 4 }}
```

**service.yaml**

```yaml
metadata:
  name: {{ include "java-web-app-chart.fullname" . }}
  labels:
{{ include "java-web-app-chart.labels" . | indent 4 }}
```

---

### include vs template

* `include` → returns a string (most commonly used)
* `template` → directly renders output

👉 **Best practice:** use `include`

---

### Benefits Summary

Using `_helpers.tpl` gives:

* DRY templates (Don’t Repeat Yourself)
* Consistent naming & labeling
* Easier upgrades and refactoring
* Production-grade Helm charts

# Helm Chart Management Models (Real‑World View)

This document explains **all major Helm chart management models used in real‑world organizations**.

---

## 1. Why So Many Helm Chart Management Models Exist

There is **no single universal way** to manage Helm charts.

Different organizations choose different models based on:

* Team size and skill level
* Number of microservices
* Dev vs Platform ownership
* Governance and security needs
* CI/CD and GitOps adoption

In reality, **most organizations evolve through multiple models over time**.

---

## Model 1: Helm Chart Inside Each Service Repository (Service‑Owned Charts)

### Structure

```text
java-order-service/
├── src/
├── Dockerfile
├── helm/
│   └── order-service/
│       ├── Chart.yaml
│       ├── values.yaml
│       └── templates/
└── README.md
```

### How This Works in Real Life

* Each microservice owns its Helm chart
* The same team maintains code, Dockerfile, and Helm chart
* CI/CD pipeline deploys directly from this repo

### Why Companies Use This

* Fast delivery
* Strong ownership ("you build it, you run it")
* Minimal coordination overhead

### Where It Fits Best

* Startups
* Small to mid‑size teams
* Early microservice adoption

### Limitations at Scale

* Helm logic duplicated across repos
* Hard to enforce standards
* Platform changes require touching many repos

---

## Model 2: Central Helm Charts Repository (Platform‑Owned Charts)

### Structure

```text
helm-charts/
├── order-service/
├── payment-service/
├── user-service/
```

### How This Works in Real Life

* Application code and Helm charts live in separate repos
* Platform / DevOps team owns Helm charts
* Developers focus only on application code

Values may live:

* In the same central repo
* Or in application repos

### Why Companies Use This

* Strong governance
* Security and compliance control
* Consistent deployment patterns

### Where It Fits Best

* Large enterprises
* Regulated environments (banking, telecom)
* Strong platform teams

### Trade‑offs

* Slower changes
* Reduced developer autonomy
* Higher coordination cost

---

## Model 3: Generic (Base) Helm Chart with Runtime Values (Template / Pages Model)

### Structure

```text
generic-app-chart/
├── Chart.yaml
├── values.yaml        # defaults
├── templates/
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── hpa.yaml
│   └── _helpers.tpl
```

Service repositories store only **values files**.

### How This Works in Real Life

* One generic chart defines *how* apps are deployed
* Service‑specific details are passed via `values.yaml`
* Same chart is reused for many services

```bash
helm upgrade --install order-service generic-app-chart -f values-prod.yaml
```

### Why Companies Use This

* Reduce Helm duplication
* Enforce standards
* Easy platform‑wide changes

### Where It Fits Best

* Growing microservice platforms
* Organizations moving toward platform engineering
* GitOps‑based setups

### Risk

* Chart can become too complex if overloaded

---

## Model 4: Helm Library Charts (Shared Logic Model)

### Structure

```text
helm-library/
├── base-app/
│   ├── Chart.yaml   # type: library
│   └── templates/
│       └── _helpers.tpl
```

### How This Works in Real Life

* Library charts contain **only reusable logic**
* They cannot be installed directly
* Service charts depend on them

### Why Companies Use This

* Share helpers, labels, naming logic
* Avoid copy‑paste
* Keep charts clean

### Where It Fits Best

* Medium to large organizations
* Teams with many similar services

---

## Model 5: GitOps‑Driven Helm Management (Deployment‑Config Repos)

### Structure

```text
app-configs/
├── order-service/
│   ├── values-dev.yaml
│   └── values-prod.yaml
```

### How This Works in Real Life

* Charts live elsewhere (service repo or generic repo)
* GitOps tool (ArgoCD / Flux) applies Helm charts
* Git becomes the source of truth

### Why Companies Use This

* Strong audit trail
* Automated sync & rollback
* Clear separation of concerns

---

## Model 6: Environment‑Centric Helm Repositories

### Structure

```text
prod/
├── order-service.yaml
├── payment-service.yaml

staging/
├── order-service.yaml
```

### How This Works in Real Life

* Helm releases grouped by environment
* Common in older or highly controlled orgs

### Trade‑off

* Simple view per environment
* Less scalable for large microservice platforms

---

## Model 7: Packaged & Versioned Helm Charts (Release Artifact Model)

### How This Works

```bash
helm package .
helm install order-service order-chart-1.2.0.tgz
```

### Why Companies Use This

* Strong version control
* Certified releases
* Controlled promotion across environments

---

## How Organizations Actually Evolve

| Stage      | Typical Model |
| ---------- | ------------- |
| Early      | Model 1       |
| Growing    | Model 3       |
| Mature     | Model 3 + 4   |
| Enterprise | Model 2 or 5  |

Most real companies **use a combination**, not just one model.

---

## Final Takeaway

* Helm chart management is an **organizational decision**, not just technical
* No model is "wrong"
* The best model depends on **scale, ownership, and maturity**

Understanding these models helps you:

* Design real‑world DevOps platforms
* Justify architecture decisions
* Think like a Platform Engineer

---

**This document represents how Helm is managed in real‑world production systems.**

# Central Helm Charts using GitHub Pages

This document explains how to host Helm charts centrally on GitHub Pages and consume them from application repositories, instead of maintaining `/helm` folders inside every service repo.

---

## 1. Problem Statement

Without a central repo:

* Each service repo has its own Helm folder
* Duplication across services
* Hard to enforce standards
* Hard to upgrade charts consistently

Goal:

* One **central Helm charts repository**
* Charts published automatically
* Services only keep `values-*.yaml`

---

## 2. Target Architecture

```
Application Repos                Central Helm Repo
------------------               ------------------
order-service/                  helm-charts/
  values-prod.yaml                charts/order-service/
  values-dev.yaml                 charts/payment-service/
                                 docs/index.yaml (GitHub Pages)
```

Helm pulls charts from GitHub Pages:

```
https://<org>.github.io/helm-charts
```

---

## 3. Repositories to Create

| Repository Name | Purpose                                          |
| --------------- | ------------------------------------------------ |
| helm-charts     | Stores all Helm charts and publishing automation |
| order-service   | Application code and environment values          |
| payment-service | Application code and environment values          |

Only `helm-charts` is new if you already have service repos.

---

## 4. Central Helm Repo Structure

After cloning `helm-charts`, create the following folders:

```
helm-charts/
├── charts/                  # All Helm charts live here
│   ├── order-service/
│   └── payment-service/
├── docs/                    # Auto-generated chart packages and index.yaml
└── .github/workflows/       # GitHub Actions
    └── release.yaml
```

---

## 5. What Goes Into Each Folder

### charts/

Each service gets a standard Helm chart:

```
charts/order-service/
├── Chart.yaml
├── values.yaml              # Defaults only
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    └── hpa.yaml
```

No environment-specific values go here.

### docs/

Generated automatically by CI:

* `order-service-1.2.0.tgz`
* `index.yaml`

Do not edit manually.

### .github/workflows/

Contains automation to publish charts.

---

## 6. GitHub Actions Workflow

Create `.github/workflows/release.yaml`

```yaml
name: Publish Helm Charts

on:
  push:
    branches: [ main ]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: azure/setup-helm@v4
      - run: mkdir -p docs
      - run: helm lint charts/*
      - run: helm package charts/* --destination docs
      - run: helm repo index docs --url https://<org>.github.io/helm-charts
      - run: |
          git config user.name github-actions
          git config user.email github-actions@github.com
          git add docs
          git commit -m "Publish charts"
          git push
```

---

## 7. What to Commit and Push

### Commit:

* `charts/*` (all chart templates)
* `.github/workflows/release.yaml`
* empty `docs/` folder (initially)

### Do NOT commit:

* secrets
* environment-specific values
* manually created `.tgz` files

---

## 8. Enable GitHub Pages

In `helm-charts` repository:

* Settings → Pages
* Source: Deploy from a branch
* Branch: main
* Folder: /docs

---

## 9. Consuming Charts in Service Repo

In each service repo:

```
order-service/
├── src/
├── Dockerfile
├── values-dev.yaml
├── values-prod.yaml
└── README.md
```

Add Helm repo:

```bash
helm repo add internal https://<org>.github.io/helm-charts
helm repo update
```

Install:

```bash
helm install order-service internal/order-service -f values-prod.yaml -n prod
```

Upgrade:

```bash
helm upgrade order-service internal/order-service -f values-prod.yaml -n prod
```

---

## 10. values-prod.yaml Example

```yaml
replicaCount: 3

image:
  repository: myrepo/order-service
  tag: 1.2.0

autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 5
```

---

## 11. Versioning Flow

| Action                           | Result            |
| -------------------------------- | ----------------- |
| Bump chart version in Chart.yaml | New package       |
| Push to main                     | Workflow runs     |
| index.yaml updated               | Chart available   |
| helm upgrade                     | Pulls new version |

---

## 12. Benefits

* Single source of truth
* No duplication
* Easy governance
* Easy rollback
* Works with CI/CD

---

## 13. Summary

Central Helm repo + GitHub Pages = scalable, clean, enterprise-grade Helm management.
