# EKS Access Guide: Endpoints, Network, and Authentication

This guide breaks down how to set up an Amazon EKS cluster so that an EC2 instance (e.g., a Bastion host or Jump box) can successfully talk to your Kubernetes API server.

-----

## 1\. Choosing the EKS Cluster Endpoint

When you create a cluster, you must decide how the "Control Plane" (the Kubernetes API) is exposed to the world.

### Option A: Public Endpoint

  * **What it is:** The API server has a public IP address and a public DNS name.
  * **Access:** Reachable via the internet.
  * **Security:** You can secure this by restricting the **CIDR ranges**.
      * *Example:* Only allow the specific Public IP of your office VPN or your EC2 instance.
  * **Best for:** Learning, development, or external CI/CD pipelines.

### Option B: Private Endpoint

  * **What it is:** The API server has *only* a private IP address inside your VPC.
  * **Access:** Reachable only from within the VPC (or connected networks like VPN/Direct Connect).
  * **Security:** Highly secure; no internet exposure.
  * **Best for:** Compliance-heavy production environments.

### Option C: Public and Private (Hybrid)

  * **What it is:** The API server has both a public and a private endpoint.
  * **Access:**
      * Traffic from *inside* the VPC (your worker nodes) uses the **Private** IP.
      * Traffic from *outside* (your laptop) uses the **Public** IP.
  * **Best for:** General production where you want easy management but efficient internal traffic.

-----

## 2\. The Network Path (EC2 → EKS)

Before authentication happens, packets must be able to flow from your EC2 instance to the EKS API.

### Scenario: Connecting to a Public Endpoint

If your cluster is Public, your EC2 instance connects via the internet (even if it's in the same AWS account).

1.  **EC2 Location:** Must have a Public IP (if in a public subnet) or access to a NAT Gateway (if in a private subnet).
2.  **Security Group (Outbound):** Must allow HTTPS (Port 443) to `0.0.0.0/0` (Internet).
3.  **EKS Allow List:** If you enabled CIDR restrictions on EKS, you must add the EC2's public IP to the allowed list.

### Scenario: Connecting to a Private Endpoint

If your cluster is Private, your EC2 instance connects via the internal VPC network.

1.  **EC2 Location:** Must be in the same VPC as the cluster (or a peered VPC).
2.  **DNS Resolution:** VPC settings `enableDnsHostnames` and `enableDnsSupport` must be `true`.
3.  **Security Groups:** The EKS Cluster Security Group must allow Inbound HTTPS (Port 443) from the EC2 instance's Security Group.

-----

## 3\. The Authentication Logic (IAM vs. RBAC)

EKS uses a two-step process to let you in.

### Step 1: Authentication (AWS IAM)

  * **"Who are you?"**
  * Kubernetes trusts AWS IAM to check your identity.
  * You run a command (via `kubectl`), which calls `aws eks get-token`.
  * AWS confirms your IAM Role/User is valid and gives you a token.

### Step 2: Authorization (Kubernetes RBAC)

  * **"What are you allowed to do?"**
  * Just because you have a valid AWS Token doesn't mean you are a Kubernetes Admin.
  * Kubernetes checks if your IAM identity is mapped to a Kubernetes Group with permissions.

> **Crucial Rule:** The IAM user/role that **created** the cluster is automatically the System Admin (`system:masters`). Everyone else must be explicitly added to the "Guest List" (the `aws-auth` ConfigMap).

-----

## 4\. The "Guest List": `aws-auth` ConfigMap

To give your EC2 instance access, you must edit a specific configuration file stored inside Kubernetes called `aws-auth`.

**Location:** Namespace `kube-system`
**Data it holds:**

1.  `mapRoles`: Maps IAM Roles → K8s Users/Groups (Used for EC2 instances/Nodes).
2.  `mapUsers`: Maps IAM Users → K8s Users/Groups (Used for humans).

### Example Configuration

Below is an example `aws-auth.yaml` that grants an EC2 instance Admin rights.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: aws-auth
  namespace: kube-system
data:
  # This section maps IAM Roles
  mapRoles: |
    # 1. Standard mapping for Worker Nodes (Required)
    - rolearn: arn:aws:iam::111122223333:role/EKS-NodeInstanceRole
      username: system:node:{{EC2PrivateDNSName}}
      groups:
        - system:bootstrappers
        - system:nodes

    # 2. YOUR EC2 INSTANCE ROLE (The "Bastion" or Admin Box)
    - rolearn: arn:aws:iam::111122223333:role/My-EC2-Admin-Role
      username: ec2-admin
      groups:
        - system:masters  # <--- This gives Full Admin Access
```

## 5\. Practical Lab: Connecting an EC2 to EKS

Here is the step-by-step workflow to make this work in real life.

### Prerequisites

  * **EKS Cluster:** Named `my-cluster` (Endpoint: Public+Private).
  * **EC2 Instance:** Running Amazon Linux 2 in the same VPC.
  * **IAM Role:** Created an IAM Role named `My-EC2-Admin-Role` and attached it to the EC2.

### Step 1: Prepare the EC2

Log into your EC2 instance via SSH and install the tools.

```bash
# Update and install AWS CLI
sudo yum update -y
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install

# Install kubectl (ensure version matches your EKS version)
curl -O https://s3.us-west-2.amazonaws.com/amazon-eks/1.29.0/2024-01-04/bin/linux/amd64/kubectl
chmod +x ./kubectl
mkdir -p $HOME/bin && cp ./kubectl $HOME/bin/kubectl && export PATH=$PATH:$HOME/bin
```

### Step 2: Configure Kubeconfig

Tell the EC2 where the cluster is. This command uses the EC2's attached IAM Role to generate the config.

```bash
aws eks update-kubeconfig --region us-east-1 --name my-cluster
```

### Step 3: The "Forbidden" Error (Expected)

Try to list nodes. It will likely fail because we haven't updated the "Guest List" yet.

```bash
kubectl get nodes
# Output: error: You must be logged in to the server (Unauthorized)
# OR: User "ec2-admin" cannot list resource "nodes" in API group "" at the cluster scope.
```

### Step 4: Update `aws-auth` (Perform this from your PC/Creator account)

You cannot do this from the EC2 yet because it has no permissions. Go to the machine where you created the cluster (your laptop or CloudShell) and edit the map.

```bash
kubectl edit configmap aws-auth -n kube-system
```

Add the section for your EC2 Role under `mapRoles`:

```yaml
    - rolearn: arn:aws:iam::111122223333:role/My-EC2-Admin-Role
      username: ec2-admin
      groups:
        - system:masters
```

### Step 5: Verify Success

Go back to your EC2 instance and run the command again.

```bash
kubectl get nodes
# Output:
# NAME                             STATUS   ROLES    AGE   VERSION
# ip-192-168-10-10.ec2.internal    Ready    <none>   5d    v1.29.0-eks...
```

**Success:** Your EC2 instance is now fully authenticated and authorized to manage the cluster.

# Understanding How You Connect to an EKS Cluster

This document explains, in a simple and natural way, **how authentication works when you connect to an Amazon EKS cluster**, what really happens behind the scenes, and **the different ways you can connect**.

---

## The Big Picture

Connecting to an EKS cluster is **not a single login step**. It happens in **two distinct layers**:

1. **AWS authentication** – Who are you in AWS?
2. **Kubernetes authentication & authorization** – What are you allowed to do in the cluster?

Many people mix these two together, which causes confusion.

---

## What `aws eks update-kubeconfig` Really Does

When you run:

```
aws eks update-kubeconfig --region us-east-1 --name my-cluster
```

This command **does NOT log you into Kubernetes**.

Instead, it:

* Authenticates you to AWS using IAM
* Fetches cluster connection details (API endpoint, CA certificate)
* Writes a kubeconfig file (~/.kube/config)
* Configures kubectl to use **IAM-based token authentication later**

Think of it as:

> “Setting up the door, not walking through it.”

---

## When Real Authentication Happens

Actual authentication happens **only when you run kubectl commands**, such as:

```
kubectl get pods
```

At that moment:

1. kubectl asks AWS for a short‑lived token
2. AWS verifies your IAM identity
3. Kubernetes validates the token
4. Kubernetes checks RBAC permissions

Only then are you truly authenticated and authorized.

---

## The Main Ways to Connect to an EKS / Kubernetes Cluster

There are **four common authentication methods**.

---

## 1. IAM‑Based Authentication (Most Common)

This is the **default and recommended** method for EKS.

How it works:

* Uses IAM users or IAM roles
* Generates short‑lived tokens
* No credentials stored in kubeconfig
* Permissions are enforced using Kubernetes RBAC

Used by:

* Developers on laptops
* EC2 instances
* CI/CD pipelines
* AWS CloudShell

This is the method used when you run `aws eks update-kubeconfig`.

---

## 2. Client Certificate Authentication

This method uses:

* X.509 client certificates
* Private keys

How it works:

* Certificate identifies the user
* Kubernetes maps the certificate identity to RBAC roles

Characteristics:

* Common in self‑managed Kubernetes
* Rare in EKS
* Hard to rotate and manage

---

## 3. ServiceAccount Authentication (Pods Only)

This method is **not for humans**.

How it works:

* Pods receive a ServiceAccount token automatically
* Token is used to authenticate to the Kubernetes API
* RBAC controls what the pod can do

Used for:

* Controllers
* Operators
* Applications running inside the cluster

---

## 4. OIDC / Federated Authentication (SSO)

This method integrates Kubernetes with external identity providers.

How it works:

* User logs in via an identity provider (SSO)
* Identity is federated
* Mapped to Kubernetes RBAC

Used by:

* Large enterprises
* Centralized identity teams

More powerful, but more complex to configure.

---
## Simple Mental Model

Think of EKS access like this:

* **AWS decides who you are** (IAM)
* **Kubernetes decides what you can do** (RBAC)
* `update-kubeconfig` prepares the path
* `kubectl` actually walks the path

## How a Tool Like Rancher Connects to ALL EKS Clusters

Now let’s apply everything above to a real-world tool like Rancher.

Rancher itself is **not a special authentication system**. It does not bypass AWS IAM or Kubernetes security. Instead, it **acts as a central client** that connects to many clusters using supported Kubernetes authentication methods.

---

## The Core Idea

When Rancher connects to multiple EKS clusters:

* **AWS IAM is still the source of truth** for identity
* **Each EKS cluster authenticates independently**
* Rancher simply automates and centralizes the connections

There is no single global EKS login.

---

## The Most Common Pattern (IAM Role + Service Account)

This is how Rancher usually connects to *all* EKS clusters in an organization.

### Step 1: Rancher Runs Somewhere

Rancher runs:

* In its own Kubernetes cluster, or
* On an EKS cluster, or
* On a VM

From Kubernetes’ point of view, Rancher is just another client.

---

### Step 2: An IAM Role Is Created per AWS Account

For each AWS account that contains EKS clusters:

* An IAM role is created
* The role has permissions like:

  * eks:DescribeCluster
  * sts:AssumeRole

This role represents **“Rancher’s identity in AWS.”**

---

### Step 3: Rancher Assumes the IAM Role

Rancher uses AWS STS to:

* Assume the IAM role
* Receive short-lived AWS credentials

This is standard AWS IAM role assumption — nothing Kubernetes-specific yet.

---

### Step 4: Rancher Gets a Kubernetes Token

For each EKS cluster:

* Rancher calls the EKS token API (same as `aws eks get-token`)
* A short-lived Kubernetes authentication token is generated

This token is **unique per cluster**.

---

### Step 5: Kubernetes RBAC Decides Access

Inside each EKS cluster:

* The IAM role is mapped (via aws-auth or EKS access entries)
* Kubernetes RBAC decides:

  * What Rancher can see
  * What Rancher can modify

This is why Rancher may have admin access in one cluster and read-only access in another.

---

## How Rancher Manages MANY Clusters

Rancher simply repeats this process:

* One IAM role → many clusters
* Or multiple IAM roles → multiple AWS accounts

Each cluster:

* Has its own API server
* Has its own RBAC
* Has its own authentication decision

Rancher **does not share credentials across clusters**.

---

## Alternative: OIDC / SSO Integration

In larger enterprises:

* Rancher integrates with an identity provider (SSO)
* Users authenticate once to Rancher
* Rancher then maps users to Kubernetes roles per cluster

Even here:

* Kubernetes still enforces RBAC
* EKS still validates tokens

Rancher just provides a unified UI and policy layer.

---

## Final One‑Line Summary

Rancher connects to all EKS clusters by assuming IAM roles, generating short-lived authentication tokens per cluster, and letting each Kubernetes cluster independently enforce RBAC. It centralizes access, but authentication and authorization remain local to each cluster.
