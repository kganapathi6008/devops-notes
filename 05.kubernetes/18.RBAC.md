# Kubernetes RBAC – Complete DevOps Guide

## 1. What is RBAC?

**RBAC (Role-Based Access Control)** is the authorization mechanism used by Kubernetes to control **who can do what** on **which resources**.

RBAC answers three questions:

1. **Who** is making the request? (User / ServiceAccount)
2. **What** are they trying to do? (verb like get, list, create)
3. **On which resource**? (pods, deployments, secrets, etc.)

RBAC is enforced by the **kube-apiserver** and applies to **every API request**.

---

## 2. Authentication vs Authorization (Important Distinction)

| Layer          | Purpose                     | Examples                   |
| -------------- | --------------------------- | -------------------------- |
| Authentication | Who are you?                | TLS certs, IAM (EKS), OIDC |
| Authorization  | What are you allowed to do? | **RBAC**                   |

RBAC **does NOT authenticate users**. It only decides permissions *after* authentication succeeds.

---

## 3. Core RBAC Objects

There are **four main RBAC resources**:

| Object             | Scope        | Purpose                           |
| ------------------ | ------------ | --------------------------------- |
| Role               | Namespace    | Define permissions in a namespace |
| ClusterRole        | Cluster-wide | Define permissions across cluster |
| RoleBinding        | Namespace    | Bind Role/ClusterRole to subject  |
| ClusterRoleBinding | Cluster-wide | Bind ClusterRole to subject       |

Think of it as:

> **Role = permissions**
> **Binding = who gets those permissions**

---

## 4. Subjects (Who Gets Access)

RBAC can grant permissions to:

### 4.1 Users

* External identities
* Examples: IAM users, OIDC users, cert-based users
* Kubernetes does **not** store user objects

### 4.2 Groups

* Logical grouping of users
* Example: `devops`, `developers`, `admins`

### 4.3 ServiceAccounts (Most Common in DevOps)

* Kubernetes-native identity
* Used by Pods, CI/CD tools, controllers

Example ServiceAccount:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: default
```

---

## 5. Verbs (What Actions Are Allowed)

Common verbs:

| Verb             | Meaning                |
| ---------------- | ---------------------- |
| get              | Read a single resource |
| list             | List resources         |
| watch            | Stream updates         |
| create           | Create new resource    |
| update           | Update entire resource |
| patch            | Partial update         |
| delete           | Delete resource        |
| deletecollection | Bulk delete            |

⚠️ Kubernetes is **verb-sensitive**. Missing `list` or `watch` often breaks controllers.

---

## 6. Resources and API Groups

Permissions apply to:

* **Resources**: pods, deployments, services, secrets
* **API Groups**: core (""), apps, batch, networking.k8s.io

Example:

```yaml
resources: ["pods"]
apiGroups: [""]
```

Deployments live in `apps` API group:

```yaml
resources: ["deployments"]
apiGroups: ["apps"]
```

---

## 7. Role (Namespace-Scoped Permissions)

A Role defines **allowed actions within a single namespace**.

### Example: Read-only access to pods

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
```

This Role:

* Applies only to `default` namespace
* Cannot access deployments or secrets

---

## 8. RoleBinding (Grant Role to Subject)

RoleBinding attaches a Role to a user, group, or ServiceAccount.

### Example: Bind Role to ServiceAccount

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: default
subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: default
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

---

## 9. ClusterRole (Cluster-Wide or Reusable Role)

ClusterRole:

* Can grant cluster-wide access
* Can also be used **inside namespaces**
* Often used for system components

### Example: Allow reading nodes (cluster-scoped)

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-reader
rules:
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "list"]
```

---

## 10. ClusterRoleBinding

Binds a ClusterRole **across the entire cluster**.

### Example: Grant node read access

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: node-reader-binding
subjects:
- kind: User
  name: jane@example.com
roleRef:
  kind: ClusterRole
  name: node-reader
  apiGroup: rbac.authorization.k8s.io
```

⚠️ Use ClusterRoleBinding carefully — it bypasses namespace isolation.

---

## 11. Role vs ClusterRole (Key Differences)

| Feature                    | Role       | ClusterRole         |
| -------------------------- | ---------- | ------------------- |
| Scope                      | Namespace  | Cluster-wide        |
| Access nodes               | ❌          | ✅                   |
| Reusable across namespaces | ❌          | ✅                   |
| Typical use                | App access | Infra / controllers |

---

## 12. Built-in ClusterRoles

Kubernetes ships with predefined roles:

| Role          | Purpose          |
| ------------- | ---------------- |
| cluster-admin | Full access      |
| admin         | Namespace admin  |
| edit          | Modify resources |
| view          | Read-only        |

Example binding:

```yaml
kind: RoleBinding
roleRef:
  kind: ClusterRole
  name: view
```

---

## 13. ServiceAccount + Pod Integration

Pods use ServiceAccounts automatically.

```yaml
apiVersion: v1
kind: Pod
spec:
  serviceAccountName: app-sa
```

Kubernetes:

* Mounts a token at `/var/run/secrets/kubernetes.io/serviceaccount`
* Uses it for API authentication

## 14. Real-World DevOps Examples

### 14.1 CI/CD Pipeline (Read & Deploy)

Permissions needed:

* get/list/watch deployments
* create/update deployments
* read pods

### 14.2 Application Pod (Least Privilege)

Permissions:

* get configmaps
* get secrets (only if required)

### 14.3 Controller

Permissions:

* list/watch resources
* update status subresources

## 15. Common RBAC Pitfalls

❌ Granting `cluster-admin` to everything
❌ Forgetting `watch` for controllers
❌ Using ClusterRoleBinding instead of RoleBinding
❌ Overusing secrets access
❌ Debugging with `kubectl` instead of `kubectl auth can-i`

---

## 16. Debugging & Verification

### Check permissions

```bash
kubectl auth can-i get pods \
  --as=system:serviceaccount:default:app-sa
```

### List RBAC objects

```bash
kubectl get roles,rolebindings -n default
kubectl get clusterroles,clusterrolebindings
```

---

## 17. Best Practices

✔ Follow **least privilege**
✔ Prefer Role + RoleBinding
✔ Reuse ClusterRoles
✔ Separate human vs machine access
✔ Version control RBAC manifests
✔ Audit permissions regularly

---

## 18. Mental Model (Final Takeaway)

Think of RBAC as:

> **IF** subject (user / SA)
> **TRIES** verb (get/list/create)
> **ON** resource (pods, secrets)
> **IN** namespace (or cluster)
> **THEN** allow or deny

Mastering RBAC is critical for secure, production-grade Kubernetes clusters.
