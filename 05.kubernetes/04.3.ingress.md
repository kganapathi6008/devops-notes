# **Understanding ALB with Kubernetes Ingress**

## **Why ALB Cannot Be Used Directly with Service YAML**

* ALB (Application Load Balancer) is **designed to work at Layer 7 (HTTP/HTTPS)**.
* Kubernetes **Service of type `LoadBalancer`** supports only **Layer 4 (TCP)** load balancers, such as Classic Load Balancer (CLB) or Network Load Balancer (NLB).
* ALB integrates with Kubernetes via the **Ingress API**, which is designed to route HTTP/S traffic based on **hostnames and paths**.

> 📌 Hence, you cannot create an ALB using only a `Service` manifest. It requires an **Ingress resource** to define the Layer 7 routing rules.

---

## **Why the ALB Ingress Controller Was Introduced**

* The native Kubernetes Ingress is a generic API that needs an **Ingress Controller** to function.
* AWS introduced the **AWS Load Balancer Controller** (formerly ALB Ingress Controller) to:

  * Automatically create and manage **Application Load Balancers (ALBs)**.
  * Translate Kubernetes Ingress objects into **ALB configurations**.
* This controller watches `Ingress` resources and provisions the necessary AWS resources dynamically.

---

## **Use Case: When to Use ALB with Ingress**

* You want **host-based or path-based routing**, e.g.,:

  * `app1.example.com` → `app1-service`
  * `example.com/app2` → `app2-service`
* You require **SSL termination** at the ALB.
* You need **native HTTP/HTTPS support** with URL rewrites, redirects, and WebSockets.
* You want to minimize the number of public load balancers (multiple services behind one ALB).

## Download and Create IAM Policy (Common for Both Approaches)

The AWS Load Balancer Controller requires a specific IAM policy that allows it to create and manage ALBs, Target Groups, Security Groups, listeners, and related EC2 and ELB resources.

Always check for the **latest recommended policy**:
[https://github.com/kubernetes-sigs/aws-load-balancer-controller/tags](https://github.com/kubernetes-sigs/aws-load-balancer-controller/tags)

```bash
# Download latest recommended IAM policy
curl -o iam-policy.json https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/main/docs/install/iam_policy.json

# Create the IAM policy
aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam_policy.json
```

### Important Note About Policy Versions

⚠️ Older policy versions may be missing required permissions.

 For example, as of version `v2.12.0`, the `iam_policy.json` from this link: [https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.12.0/docs/install/iam_policy.json](https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.12.0/docs/install/iam_policy.json) is missing the `ec2:DescribeRouteTables` permission. If you see errors like:

```
AccessDenied: ec2:DescribeRouteTables
```

Manually add the following statement to the policy:

```json
{
  "Effect": "Allow",
  "Action": "ec2:DescribeRouteTables",
  "Resource": "*"
}
```

This applies **regardless of which approach you choose** below.

## IAM Roles for Service Accounts (IRSA) – Two Supported Approaches

The AWS Load Balancer Controller uses **IRSA** to obtain AWS credentials securely. There are **two valid ways** to configure this.

Both approaches result in the **same final outcome**:

* A Kubernetes ServiceAccount annotated with an IAM Role
* The controller pod assumes that role via OIDC

---

# Approach 1 – Automated IRSA Using `eksctl` (Recommended)

This is the simplest and safest approach. `eksctl` automatically:

* Creates an IAM role
* Configures the OIDC trust policy
* Attaches the IAM policy
* Annotates the Kubernetes ServiceAccount

### Create IRSA and Attach Policy via eksctl

```bash
eksctl create iamserviceaccount \
  --cluster=${cluster_name} \
  --namespace=kube-system \
  --name=aws-load-balancer-controller \
  --attach-policy-arn=arn:aws:iam::${account_id}:policy/AWSLoadBalancerControllerIAMPolicy \
  --override-existing-serviceaccounts \
  --region ${region} \
  --approve
```

### What eksctl Does Internally

* Creates an IAM role with `sts:AssumeRoleWithWebIdentity`
* Trusts the EKS OIDC provider
* Restricts the role to:
  `system:serviceaccount:kube-system:aws-load-balancer-controller`
* Annotates the ServiceAccount with the role ARN

---

### Verify IAM Service Account

```bash
kubectl get sa -n kube-system
kubectl describe sa aws-load-balancer-controller -n kube-system
```

Example output:

```text
Name:                aws-load-balancer-controller
Namespace:           kube-system
Labels:              app.kubernetes.io/managed-by=eksctl
Annotations:
  eks.amazonaws.com/role-arn: arn:aws:iam::<account-id>:role/eksctl-eks-dev-env-addon-iamserviceaccount-ku-Role1
```

# Approach 2 – Manual IAM Role + Service Account (Enterprise / Audited Setup)

This approach is functionally identical but gives **full control** over IAM resources. It is commonly used in regulated or enterprise environments.

### Step 1: Create IAM Role with OIDC Trust Policy

The trust policy must reference:

* Your EKS OIDC provider
* The correct namespace and ServiceAccount name

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/oidc.eks.<region>.amazonaws.com/id/<OIDC_ID>"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.<region>.amazonaws.com/id/<OIDC_ID>:sub": "system:serviceaccount:kube-system:aws-load-balancer-controller"
        }
      }
    }
  ]
}
```

⚠️ Any mismatch in **namespace** or **service account name** will prevent role assumption.

---

### Step 2: Attach IAM Policy to the Role

```bash
aws iam attach-role-policy \
  --role-name AWSLoadBalancerControllerRole \
  --policy-arn arn:aws:iam::${account_id}:policy/AWSLoadBalancerControllerIAMPolicy
```

---

### Step 3: Create Kubernetes ServiceAccount Manually

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: aws-load-balancer-controller
  namespace: kube-system
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::<ACCOUNT_ID>:role/AWSLoadBalancerControllerRole
```

```bash
kubectl apply -f aws-load-balancer-controller-sa.yaml
```

## **Sample Example Ingress Manifest for ALB**

```yaml
# 1. Pod Definition
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx-server
spec:
  containers:
  - name: nginx-container
    image: nginx:latest
    ports:
    - containerPort: 80
---
# 2. Service Definition
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  selector:
    app: nginx-server
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
---
# 3. Ingress Definition for ALB
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nginx-ingress
  annotations:
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
spec:
  ingressClassName: alb
  rules:
  - http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: nginx-service
            port:
              number: 80

```

## **Delete the ALB Ingress and Related Resources**

To clean up the ALB and all related AWS resources created via the AWS Load Balancer Controller:

### **Step 1: Delete the Ingress Resource**

```bash
kubectl delete ingress my-ingress
```

> This will trigger the AWS Load Balancer Controller to delete the associated ALB.

### **Step 2: Uninstall the Helm Chart**

```bash
helm uninstall aws-load-balancer-controller -n kube-system
```

### **Step 3: Delete the IAM Service Account**

```bash
eksctl delete iamserviceaccount \
  --cluster=${cluster_name} \
  --namespace=kube-system \
  --name=aws-load-balancer-controller \
  --region=${region}
```

> ✅ This command will remove the Kubernetes service account and delete the associated IAM role created by `eksctl`.

### **Step 4: Delete the IAM Policy (Optional)**

```bash
aws iam delete-policy \
  --policy-arn arn:aws:iam::${account_id}:policy/AWSLoadBalancerControllerIAMPolicy
```

> Note: Ensure no other IAM roles are using this policy before deletion.

# Types of Routing in Ingress

Kubernetes Ingress supports multiple routing mechanisms via annotations and rules. The two most commonly used types are:

* **Host-based routing**: Routes traffic based on the domain/subdomain.
* **Path-based routing**: Routes traffic based on the URL path under the domain.


In Kubernetes, an **Ingress** is an API object that manages external access to services in a cluster, typically HTTP. AWS provides the **Application Load Balancer (ALB)** as a Layer 7 load balancer that can intelligently route traffic based on rules. When using the **AWS Load Balancer Controller**, you can configure ALB with both **host-based** and **path-based** routing using annotations in your Ingress resources.

These routing mechanisms are especially useful when you want to:

* Host **multiple applications** on a single ALB.
* Route traffic to different backends **based on request URLs** or **hostnames**.

---

## Why We Need Host-Based and Path-Based Routing

### Without Routing:

If you create a `Service` of type `LoadBalancer`, each service gets its **own ELB**. This is:

* Costly (as you pay for each Load Balancer)
* Harder to manage (especially for many microservices)

### With ALB + Ingress:

* Use **one ALB** to handle multiple routes.
* Route requests to different services **based on domain (host)** or **URL path**.
* SSL termination and centralized traffic control.

---

## Host-Based Routing

**Use Case**: You have different applications and want each accessible under a different subdomain.

* `app1.example.com` → forwards to `app1-service`
* `app2.example.com` → forwards to `app2-service`

**Example Ingress YAML:**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: host-routing-ingress
  annotations:
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS": 443}]'
    alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:<region>:<account-id>:certificate/<cert-id>
    alb.ingress.kubernetes.io/ssl-redirect: '443'
spec:
  ingressClassName: alb
  rules:
  - host: app1.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app1-service
            port:
              number: 80
  - host: app2.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app2-service
            port:
              number: 80
```

---

## Path-Based Routing

**Use Case**: You want to route requests based on path under the same domain.

* `example.com/app1/` → forwards to `app1-service`
* `example.com/app2/` → forwards to `app2-service`

**Example Ingress YAML:**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: path-routing-ingress
  annotations:
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS": 443}]'
    alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:<region>:<account-id>:certificate/<cert-id>
    alb.ingress.kubernetes.io/ssl-redirect: '443'
spec:
  ingressClassName: alb
  rules:
  - host: example.com
    http:
      paths:
      - path: /app1/*
        pathType: Prefix
        backend:
          service:
            name: app1-service
            port:
              number: 80
      - path: /app2/*
        pathType: Prefix
        backend:
          service:
            name: app2-service
            port:
              number: 80
```

---

## Summary

| Routing Type | Use Case                     | Example Host/Path                      |
| ------------ | ---------------------------- | -------------------------------------- |
| Host-based   | Different apps per subdomain | `app1.example.com`, `app2.example.com` |
| Path-based   | Different apps per path      | `example.com/app1/`, `example.com/app2/` |

---

## Real-World Examples: Google Services

### ✅ Host-based routing:

In host-based routing, the ALB routes traffic based on the domain (host) in the request.

| URL                                                          | Host                | Route Example            |
| ------------------------------------------------------------ | ------------------- | ------------------------ |
| [https://drive.google.com/](https://drive.google.com/)       | drive.google.com    | Route to Google Drive    |
| [https://mail.google.com/](https://mail.google.com/)         | mail.google.com     | Route to Gmail           |
| [https://calendar.google.com/](https://calendar.google.com/) | calendar.google.com | Route to Google Calendar |
| [https://docs.google.com/](https://docs.google.com/)         | docs.google.com     | Route to Google Docs     |

These are routed based on different subdomains, so this is **host-based routing**.

### ✅ Path-based routing:

In path-based routing, the ALB routes traffic based on the path after the domain.

| URL                                                                            | Host              | Path           | Route Example          |
| ------------------------------------------------------------------------------ | ----------------- | -------------- | ---------------------- |
| [https://docs.google.com/presentation/](https://docs.google.com/presentation/) | *docs.google.com* | /presentation/ | Route to Google Slides |
| [https://docs.google.com/spreadsheets/](https://docs.google.com/spreadsheets/) | docs.google.com   | /spreadsheets/ | Route to Google Sheets |

These are routed based on different paths under the same domain, so this is **path-based routing**.

---

## Conclusion

By using ALB Ingress with host-based or path-based routing, you simplify your infrastructure, reduce cost, and increase flexibility in managing multiple services. This approach is ideal for microservices, dev/test environments, and scalable production-grade architectures.

Make sure to use **valid ACM certificates**, **correct DNS records in Route 53**, and update ingress rules to match your application's structure.

## Path-Based Routing and Base Path Rewrite (Simple Explanation)

Think of **path-based routing** like a traffic sign that looks at the **URL path** and decides where to send the request.

For example, when a user opens:

```
https://example.com/app1
```

Kubernetes Ingress does only one job here:
👉 it matches `/app1` and forwards the request to the correct service.

**Important part:** Ingress does **not** change the request path.

So the backend application receives:

```
GET /app1
```

---

### Why this becomes a problem

Most simple apps (like default nginx or apache containers) are very basic. They are configured to serve content only from the root path:

```
/
```

They work perfectly when they receive:

```
GET /
```

But when they receive:

```
GET /app1
```

They don’t know what `/app1` means — so they respond with **404 Not Found**.

At this point:

* Ingress is working
* Service is working
* Pod is running fine
* ❌ The application itself can’t handle the path

So the failure feels confusing, but nothing is actually broken in Kubernetes.

---

### What “Base Path Rewrite” really means

Base path rewrite is simply **removing the prefix** before the request reaches the application.

In simple terms:

```
User sends:     /app1
Ingress sends:  /
```

Now the application sees exactly what it expects and starts responding normally.

---

### When rewrite is NOT needed

You do NOT need a rewrite if:

* The application is designed to work with subpaths
* The app explicitly knows routes like `/app1`, `/app2`

This is common for APIs and modern backend frameworks.

---

### Practical recommendation (real-world)

In real production setups:

* ✅ Use **host-based routing** whenever possible (`app1.example.com`)
* ❌ Avoid path-based routing for basic nginx/apache containers

Host-based routing keeps URLs clean and avoids rewrite complexity altogether.

---

### One-line takeaway

Path-based routing works, but your **application must understand the path**. If it doesn’t, you either rewrite the path or switch to host-based routing.

## Multiple Ingress Files and Load Balancers (Simple Explanation)

It’s easy to assume that **more Ingress YAML files automatically mean more Load Balancers**, but that’s not how it works.

An Ingress file is just a **set of routing rules**. The AWS Load Balancer Controller looks at those rules and decides whether it can place them on an **existing ALB** or needs to create a new one.

By default, when multiple Ingress resources:

* use the same ingress class (`alb`)
* live in the same namespace
* use the same scheme (internet-facing or internal)

They are **merged together behind a single Application Load Balancer**.

So even if you have:

* one Ingress for app1
* another Ingress for app2

They usually end up sharing **one ALB**, with different host or path rules.

---

### When a new Load Balancer is created

A separate Load Balancer is created only when you **intentionally separate traffic**, for example:

* you explicitly define different ALB groups
* you mix internal and internet-facing schemes
* you use different ingress controllers

In those cases, AWS cannot reuse the same ALB, so a new one is provisioned.

---

### How to think about it

A simple mental model:

* **Ingress = routing rules**
* **ALB = shared entry point**

Multiple rule files can point to the **same entry point**, unless you clearly say otherwise.

---

### Key takeaway

Creating multiple Ingress resources does **not** automatically create multiple Load Balancers.

A new ALB appears only when you explicitly design your setup to require one.

---

## Do Ingress resources use a default group?

Yes — you can think of it this way:

If you **do not specify any group**, the AWS Load Balancer Controller automatically treats all compatible Ingress resources as part of a **default (implicit) group**.

This means:

* You do **not** need to define a group explicitly
* Ingress resources that look compatible are **automatically combined**
* They end up sharing the **same Application Load Balancer**

So in practice, it behaves like there is a **default group**, even though you never write it in YAML.

---

### What “compatible” really means

Ingress resources will naturally join this default group when they:

* use the same `ingressClassName` (for example `alb`)
* are in the same namespace
* use the same scheme (internet-facing or internal)

When these conditions match, the controller simply says:

> “I can place all these rules on one ALB.”

---

### When you need to define a group explicitly

You only define a group when you want **separation**.

```bash
annotations:
  alb.ingress.kubernetes.io/group.name: my-shared-group
```

For example:

* you want a dedicated ALB for a specific app
* you want to isolate traffic for security or cost reasons

In that case, you explicitly name a group and the controller creates a **separate ALB**.

---

### Simple way to remember

* No group defined → **shared (default) ALB**
* Group defined → **separate ALB**

---

### Final takeaway

You don’t have to configure anything special to share a Load Balancer.

Sharing is the **default behavior** — separation is something you opt into intentionally.
