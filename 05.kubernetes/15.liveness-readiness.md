# Kubernetes Probes: Liveness, Readiness, and Startup

This document provides a clear, practical explanation of Kubernetes **liveness**, **readiness**, and **startup** probes—what they are, how they work, real-world scenarios, example YAML, and what happens when each probe fails.

---

## **1. Why Probes Exist in Kubernetes**

Containers may start, run, and fail in unexpected ways. Kubernetes needs a way to check **if your container is healthy**, **ready to receive traffic**, or **has fully started**.

To manage this, Kubernetes uses **three probes**:

* **Liveness Probe** → "Is the container *alive*?"
* **Readiness Probe** → "Is the container *ready to serve traffic*?"
* **Startup Probe** → "Has the application *finished starting*?"

Each probe protects your application in different ways.

---

# **2. Liveness Probe**

### **Purpose**

A liveness probe checks whether the container is **still running properly**. If this probe fails, Kubernetes assumes the application is stuck or dead and will **restart the container**.

### **When to Use**

Use liveness probes when:

* Your application may get stuck (deadlocks)
* Your app has long-running connections that may freeze
* Memory leaks cause it to hang
* A bug might cause the server to stop responding

### **Real-World Examples**

1. **Java app deadlock** → JVM running but app is frozen.
2. **Node.js event loop blocked** → process alive but not responding.
3. **Nginx stuck due to config reload or failed upstream**.

### **What happens when liveness probe fails?**

✔ Kubernetes **restarts the container** (using the pod’s restart policy).
✘ Container is marked **Unhealthy**, not Ready.

### **YAML Example**

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: liveness-port
  failureThreshold: 3
  initialDelaySeconds: 5
  timeoutSeconds: 2
  periodSeconds: 10
  terminationGracePeriodSeconds: 5
```

### **Fields Explained**

* **initialDelaySeconds** → How long to wait before first probe (startup buffer)
* **periodSeconds** → How often to probe
* **timeoutSeconds** → Probe timeout
* **failureThreshold** → Fail X times before marking container unhealthy
* **terminationGracePeriodSeconds** → How long to wait before force-killing container on failure

---

# **3. Readiness Probe**

### **Purpose**

A readiness probe checks if the container is **ready to receive traffic**.

### **When to Use**

Use readiness probes when your app:

* Takes time to warm up (loading ML models, caching, migrations)
* Should not receive traffic until dependencies are ready (DB, API, message queues)
* Needs graceful shutdown during rolling upgrades

### **Real-World Examples**

1. **Spring Boot app needs 20s before serving HTTP traffic**.
2. **API must connect to PostgreSQL before handling requests**.
3. **Microservice checks Kafka readiness before consuming messages**.

### **What happens when readiness probe fails?**

✔ Pod is marked **NotReady**
✔ Removed from service endpoints
✘ **Container is NOT restarted**
✔ Traffic stops going to this pod

This prevents outages during partial failures.

### **YAML Example**

```yaml
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 3
  periodSeconds: 5
  timeoutSeconds: 2
  failureThreshold: 3
```

---

# **4. Startup Probe**

### **Purpose**

Used to check **slow-starting applications**. Prevents liveness probe from killing an app before it has finished starting.

### **When to Use**

Use startup probes when your app:

* Takes **very long** to start (e.g., large frameworks)
* Loads huge data sets or caches
* Needs DB migrations before starting
* Runs on slow hardware

### **Real-World Examples**

1. **Large Java monolith (30–60 sec startup)**.
2. **Python ML service loading 2GB model at startup**.
3. **Redis cluster warming up memory**.

### **What happens when startup probe fails?**

✔ Container is restarted
✔ Liveness & readiness probes remain **disabled** until startup passes

### **YAML Example**

```yaml
startupProbe:
  httpGet:
    path: /startup
    port: 8080
  failureThreshold: 30
  periodSeconds: 5
```

Here: 30 × 5 = **150 seconds allowed for startup**.

---

# **5. Differences Between Liveness, Readiness, Startup**

| Probe         | Purpose                        | Failure Effect                   | When Used                               |
| ------------- | ------------------------------ | -------------------------------- | --------------------------------------- |
| **Liveness**  | Is container alive?            | Restart container                | Detect hangs, deadlocks, freezes        |
| **Readiness** | Can container receive traffic? | Remove from service, NOT restart | App warming up or temporarily unhealthy |
| **Startup**   | Has app started fully?         | Restart container                | Slow starting apps                      |

---

# **6. What Happens if Probes Fail?**

## **If Liveness Probe Fails:**

* Pod is marked **Unhealthy**
* Container is **restarted**
* New liveness cycle begins

Example scenario:

> Node.js event loop freezes → /healthz never responds → Kubernetes restarts it

---

## **If Readiness Probe Fails:**

* Pod is **removed from load balancer / service endpoints**
* Pod continues running
* No restart occurs

Example scenario:

> DB connection lost → App cannot serve traffic → Pod set to NotReady → Traffic stops

---

## **If Startup Probe Fails:**

* Container is restarted
* Liveness/Readiness probes remain **disabled** until startup succeeds

Example scenario:

> App takes too long to load a model → Startup fails → Container restarts with more time

---

# **7. Combined Probe Example (Best Practice)**

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5

startupProbe:
  httpGet:
    path: /startup
    port: 8080
  failureThreshold: 30
  periodSeconds: 5
```

---

# **8. Real-World Use Cases Summary**

### **Use Liveness Probe When:**

* Your app may freeze or deadlock
* Long-running server may stop responding
* You want Kubernetes to self-heal stuck containers

### **Use Readiness Probe When:**

* App needs time to warm up
* App must stop receiving traffic during failures
* You want zero-downtime rolling updates

### **Use Startup Probe When:**

* App startup time is unpredictable or long
* Liveness probe would kill app prematurely
* Large monoliths / ML apps

---

# **9. Conclusion**

* **Readiness Probe = Traffic control**
* **Liveness Probe = Self-healing / restart**
* **Startup Probe = Prevent premature restarts for slow apps**

Together, these probes make Kubernetes applications reliable, fault-tolerant, and production-ready.

# **10. Extended Explanation and Deep-Dive Topics**

To further strengthen your understanding after point 9, here are deeper explanations of how probes behave in production, operational scenarios, and additional best practices.

## **10.1 Failure Simulation Scenarios**

### **Liveness Probe Failure Scenario**

* Your service code enters a deadlock.
* `/healthz` stops responding.
* Liveness probe fails 3 times.
* Kubernetes **restarts** the container.
* Traffic continues to flow to other healthy pods.

### **Readiness Probe Failure Scenario**

* Application temporarily loses DB connection.
* `/ready` endpoint returns 500 or times out.
* Readiness probe fails.
* Pod is **removed from Service endpoints** but **NOT restarted**.
* Traffic flows only to healthy pods.
* When DB is restored, readiness passes and pod joins again.

### **Startup Probe Failure Scenario**

* App takes too long to initialize (e.g., large ML model).
* Startup probe keeps checking `/startup`.
* If app takes longer than the allowed time → container restarts.
* Liveness probe never kills the app prematurely.

## **10.2 Best Practices for Each Probe**

### **Liveness Probe Best Practices**

* Do NOT point liveness to expensive operations.
* Keep `/healthz` **very lightweight**.
* Avoid using readiness endpoints for liveness—they serve different purposes.
* Use high `failureThreshold` if app may be slow under load.

### **Readiness Probe Best Practices**

* Ensure `/ready` checks dependencies (DB, cache, messaging).
* Use readiness gates during rolling deployments.
* Does NOT need to indicate deep application health—only traffic readiness.

### **Startup Probe Best Practices**

* Always use startup probes for slow-starting apps (Java, ML, etc.).
* Set long `failureThreshold × periodSeconds` to match realistic startup time.
* Avoid combining long waits in liveness probe—startup probe is the right tool.

## **10.3 Diagram (Text-Based)**

```
             +----------------------+
             | Container Starts     |
             +----------+-----------+
                        |
                Startup Probe Runs
                        |
     +------------------+-------------------+
     |                                      |
Startup FAIL                          Startup PASS
     |                                      |
Container Restarted                 Liveness & Readiness Enabled
                                        |
                               +---------+---------+
                               |                   |
                        Liveness OK         Liveness FAIL
                               |                   |
                        Pod is Alive       Container Restarted

```

## **10.4 Summary of Extended Concepts**

* Startup probe protects slow applications from liveness probe restarts.
* Readiness probe controls traffic flow but never restarts the pod.
* Liveness probe ensures the app self-heals by restarting frozen containers.

# 11. Complete Working YAML Example Using Public Image (Agnhost)

Below is a real, fully working Kubernetes Deployment + Service you can use to test **liveness**, **readiness**, and **startup** probes. It uses the public image:

**Image:** `registry.k8s.io/e2e-test-images/agnhost:2.40`

This image exposes built-in endpoints:

* `/healthz` → used for **liveness** and **startup** probes
* `/readyz` → used for **readiness** probe

These behave exactly like real-world applications, making the image perfect for learning and testing.

---

## **11.1 Complete YAML Manifest**

Use this YAML as-is:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: probes-agnhost-deployment
  labels:
    app: probes-agnhost
spec:
  replicas: 2
  selector:
    matchLabels:
      app: probes-agnhost
  template:
    metadata:
      labels:
        app: probes-agnhost
    spec:
      containers:
      - name: agnhost
        image: registry.k8s.io/e2e-test-images/agnhost:2.40
        args:
          - netexec
          - --http-port=8080
        ports:
        - name: http
          containerPort: 8080

        startupProbe:
          httpGet:
            path: /healthz
            port: http
          periodSeconds: 5
          failureThreshold: 24
          timeoutSeconds: 2

        readinessProbe:
          httpGet:
            path: /readyz
            port: http
          initialDelaySeconds: 5
          periodSeconds: 5
          timeoutSeconds: 2
          failureThreshold: 3
          successThreshold: 1

        livenessProbe:
          httpGet:
            path: /healthz
            port: http
          initialDelaySeconds: 10
          periodSeconds: 10
          timeoutSeconds: 2
          failureThreshold: 3
          terminationGracePeriodSeconds: 5

        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "500m"
            memory: "256Mi"

---
apiVersion: v1
kind: Service
metadata:
  name: probes-agnhost-service
  labels:
    app: probes-agnhost
spec:
  selector:
    app: probes-agnhost
  ports:
  - name: http
    port: 80
    targetPort: 8080
  type: ClusterIP
```

---

## **11.2 Explanation of How This Works**

### **1. Image & Command**

* The container runs: `agnhost netexec --http-port=8080`
* This starts an HTTP server providing:

  * `/healthz` → healthy/unhealthy
  * `/readyz` → ready/not ready

These endpoints automatically simulate success/failure like a real microservice.

---

## **2. Startup Probe Explanation**

```yaml
startupProbe:
  httpGet:
    path: /healthz
    port: http
  periodSeconds: 5
  failureThreshold: 24
```

* Kubernetes gives the application **up to 120 seconds** (24 × 5) to start.
* Until startup succeeds:

  * **readiness** is disabled
  * **liveness** is disabled
* If startup probe fails too long → **container restarts**.

This protects slow-starting applications.

---

## **3. Readiness Probe Explanation**

```yaml
readinessProbe:
  httpGet:
    path: /readyz
    port: http
```

* Controls whether the pod receives traffic.
* If it fails:

  * Pod is marked **NotReady**
  * Removed from Service endpoints
  * **No restart occurs**

Perfect for DB-dependent apps or apps warming up.

---

## **4. Liveness Probe Explanation**

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: http
```

* Checks if the app is alive.
* If `/healthz` fails:

  * Kubernetes marks the container **Unhealthy**
  * Restarts it

This is used to detect deadlocks, freezes, and stuck containers.
