# Cluster Autoscaler on Amazon EKS

This is a clear, production-ready guide for installing Cluster Autoscaler (CA) on Amazon EKS using the official auto-discovery YAML.

---

## Quick TL;DR

- Cluster Autoscaler (CA) adds or removes worker nodes when pods can or cannot be scheduled.
- CA needs two permission sets:
  - Kubernetes RBAC (ServiceAccount + ClusterRole)
  - AWS IAM (either via node IAM role in legacy setups, or via IRSA — recommended)
- Use auto-discovery with ASG tags so CA finds node groups automatically.
- CA's version must match your Kubernetes minor version (e.g., Kubernetes 1.32 → CA v1.32.x).
- Always tag ASGs / node groups so CA can manage them.

## 1. What is Cluster Autoscaler?

Cluster Autoscaler is a Kubernetes component that changes the number of worker nodes:

- Scales up: when pods are Pending because there’s no space to schedule them.
- Scales down: when nodes are underused and pods can move to other nodes.

What CA does NOT do:

- It does not scale pods (use HPA for that).
- It does not use metrics-server to decide scaling.
- It does not look at CPU or memory usage directly.

---

## 2. Why you need it on EKS

EKS does not automatically change the node count. If you only use HPA to scale pods, pods may stay Pending because there is no additional node capacity. CA makes nodes available when pods are Pending.

Flow with CA:
- HPA increases pod replicas → some pods Pending → CA adds nodes → pods get scheduled.

---

## 3. How Cluster Autoscaler works (simple)

CA constantly checks the cluster for Pending/unschedulable pods. It:

1. Sees Pending pods.
2. Tries to simulate scheduling.
3. Decides which node group (ASG) can host the pods.
4. Calls AWS to increase the ASG desired capacity.
5. Later, removes nodes that are safe to remove.

Important: CA reacts to scheduler failures (pods Pending), not to resource metrics.

---

## 4. What Cluster Autoscaler needs to function

CA needs two separate permission sets:

a) Kubernetes permissions (RBAC)
- To watch Pods, Nodes, and PodDisruptionBudgets
- To evict pods during scale-down
- Leader election, etc.
- Implemented with a ServiceAccount + ClusterRole (+ ClusterRoleBinding)

b) AWS permissions (IAM)
- To discover ASGs, increase desired capacity, and terminate instances
- Two ways to provide IAM permissions:
  - Legacy: node IAM role (IMDS) — not recommended
  - IRSA (IAM Roles for Service Accounts) — recommended (least privilege)

---

## 5. Legacy model (NOT recommended)

How it works
- CA pod runs on an EC2 node and uses the node's IAM role via the Instance Metadata Service (IMDS).
- The node IAM role must have autoscaling permissions.

How to set it up (summary):
1. Create an IAM policy with autoscaling actions.
2. Attach the policy to the node group IAM role (Managed Node Group role or EC2 instance role).
3. Apply the upstream YAML (no IRSA annotation needed).

Why this is not recommended
- All pods on that node share the same AWS permissions.
- Larger security risk (bigger blast radius).
- Not least-privilege.

---

## 6. IRSA model (RECOMMENDED)

IRSA gives AWS permissions to a single Kubernetes ServiceAccount, not to nodes. This is the preferred production pattern.

Overview
- Create an AWS IAM role that allows Cluster Autoscaler to read and modify Auto Scaling Groups.
- Allow the EKS cluster’s OIDC provider to assume the IAM role using IRSA.
- Attach the IAM role to the Cluster Autoscaler ServiceAccount via annotation.
- Deploy CA using that ServiceAccount → Run the Cluster Autoscaler pod using the annotated ServiceAccount.
- Use ASG tag-based auto-discovery → Enable Cluster Autoscaler to discover and manage node groups based on ASG tags.

**Step-by-step (simple)**

**Step 1 — Enable OIDC for the cluster (one time)**
- This allows pods to assume IAM roles.
- Using eksctl:
  ```
  eksctl utils associate-iam-oidc-provider \
    --cluster <My_Cluster_Name> \
    --approve
  ```
- Or follow AWS Console / CLI instructions to add the OIDC provider.

**Step 2 — Required ASG tags**
Every ASG or node group you want CA to manage must have these tags:
```
k8s.io/cluster-autoscaler/enabled = true
k8s.io/cluster-autoscaler/<My_Cluster_Name> = owned
```
If tags are missing, CA will not scale that ASG.

**Step 3 — Least-privilege IAM policy (example)**
This policy allows scale changes only for ASGs that have the required tags, and allows read actions needed by CA.

Example policy JSON:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "autoscaling:SetDesiredCapacity",
        "autoscaling:TerminateInstanceInAutoScalingGroup"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/k8s.io/cluster-autoscaler/enabled": "true",
          "aws:ResourceTag/k8s.io/cluster-autoscaler/<My_Cluster_Name>": "owned"
        }
      }
    },
    {
      "Effect": "Allow",
      "Action": [
        "autoscaling:DescribeAutoScalingGroups",
        "autoscaling:DescribeAutoScalingInstances",
        "autoscaling:DescribeLaunchConfigurations",
        "autoscaling:DescribeScalingActivities",
        "autoscaling:DescribeTags",
        "ec2:DescribeImages",
        "ec2:DescribeInstanceTypes",
        "ec2:DescribeLaunchTemplateVersions",
        "ec2:GetInstanceTypesFromInstanceRequirements",
        "eks:DescribeNodegroup"
      ],
      "Resource": "*"
    }
  ]
}
```
Notes:
- Replace `<My_Cluster_Name>` with your cluster name.
- The Describe permissions are read-only and kept broad because CA needs to read ASG and instance info.

**Step 4 — Create IAM role for Cluster Autoscaler**
- Create an IAM role and attach the policy above.
- The role trust policy should allow the cluster OIDC provider to assume the role and should restrict it to the CA ServiceAccount only.

Example trust policy:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/oidc.eks.<region>.amazonaws.com/id/<OIDC_ID>"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.<region>.amazonaws.com/id/<OIDC_ID>:sub": "system:serviceaccount:kube-system:cluster-autoscaler"
        }
      }
    }
  ]
}
```
Replace:
- `<ACCOUNT_ID>` — your AWS account ID
- `<region>` — e.g. us-east-1
- `<OIDC_ID>` — the OIDC provider ID for your cluster (shown when you associate OIDC)
- The `sub` condition ties the role to `kube-system:cluster-autoscaler` ServiceAccount only.

**Step 5 — Create the Kubernetes ServiceAccount**
Apply this ServiceAccount in kube-system and annotate it with the role ARN:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: cluster-autoscaler
  namespace: kube-system
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::<ACCOUNT_ID>:role/ClusterAutoscalerRole
```

**Step 6 — Deploy Cluster Autoscaler (use the upstream YAML)**

Make sure the Deployment uses `serviceAccountName: cluster-autoscaler`.

Recommended auto-discovery argument (in the CA deployment):
```
--node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/<My_Cluster_Name>
```
This makes CA discover all ASGs that match the tags.

Important version rule
- CA must match the Kubernetes minor version of your cluster. For example, Kubernetes 1.32 should use Cluster Autoscaler v1.32.x.
- Always check the CA release page when you install (image name may change).

**Example snippet from a Deployment:**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cluster-autoscaler
  namespace: kube-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: cluster-autoscaler
  template:
    metadata:
      labels:
        app: cluster-autoscaler
    spec:
      serviceAccountName: cluster-autoscaler
      priorityClassName: system-cluster-critical
      containers:
      - name: cluster-autoscaler
        image: registry.k8s.io/autoscaling/cluster-autoscaler:v1.32.1
        command:
        - ./cluster-autoscaler
        - --cloud-provider=aws
        - --v=4
        - --stderrthreshold=info
        - --expander=least-waste
        - --skip-nodes-with-local-storage=false
        - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/<My_Cluster_Name>
        resources:
          requests:
            cpu: 100m
            memory: 600Mi
          limits:
            cpu: 100m
            memory: 600Mi
```
Notes:
- Replace the image tag with the correct CA version for your cluster minor version.
- Replace `<My_Cluster_Name>`.

---

## 7. Explanation of Key Flags (simple)

- --cloud-provider=aws
  Use AWS as cloud provider.

- --node-group-auto-discovery=asg:tag=...
  Discover node groups (ASGs) using tags.

- --expander=least-waste
  Choose the node group that wastes the least resources (cost-efficient).

- --skip-nodes-with-local-storage=false
  Allow CA to consider nodes with local storage for scale-down (default behavior can avoid nodes with local storage).

- --v=4
  Logging verbosity. v4 is good for production debugging.

## 8. Verifying the installation

Check pod:
```
kubectl get pods -n kube-system | grep cluster-autoscaler
```

Check logs:
```
kubectl logs -n kube-system deployment/cluster-autoscaler
```
Expected log lines on scale-up:
```
Scale-up: setting group <asg-name> desired size to X
```

If you see AWS API permission errors in logs, check the IAM role and IRSA setup.

---

## 9. When Cluster Autoscaler will NOT scale

- No Pending pods → nothing to do.
- ASG maxSize reached → cannot increase further.
- ASG missing required tags → CA does not discover it.
- Pods have impossible constraints (nodeSelector, nodeAffinity) → no matching nodes.
- Missing IAM permissions → AWS calls fail.
- PDBs block evictions → scale-down is blocked.

---

## 10. Scale‑down rules (when CA will NOT remove a node)

CA will not remove a node if:

- Pods on the node are not replicated (would lose unique pods).
- Pods use local storage that cannot be moved.
- PodDisruptionBudgets (PDBs) prevent eviction.
- Node runs critical system pods (kube-system, daemonsets, etc).

---

## 11. Production best practices (short)

- Always set resource requests for pods.
- Use multiple node groups with different instance types / sizes.
- Set realistic ASG maxSize and minSize.
- Monitor CA logs and test scaling regularly.
- Use IRSA (least privilege) — do not grant autoscaling permissions to all nodes.
- Avoid single-node clusters for production.

---

## 12. Final short summary

- Cluster Autoscaler scales worker nodes in response to Pending pods.
- It needs Kubernetes RBAC + AWS IAM.
- Legacy model uses node IAM role (not recommended). IRSA is recommended.
- Use ASG tag auto-discovery for easy management.
- Match CA version to your Kubernetes minor version.
- Tag your ASGs correctly and give CA the right IAM permissions.
