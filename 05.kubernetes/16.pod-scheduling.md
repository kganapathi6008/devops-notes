# Kubernetes Pod Scheduling Guide

This document explains all the ways you can control **where** Kubernetes schedules your Pods.

---

## **Overview**

Kubernetes does not usually let you pick a specific node for each Pod manually. Instead, you give **constraints**, **preferences**, or **permissions**, and the Kubernetes Scheduler chooses the best node.

There are multiple mechanisms to influence scheduling:

1. Hard pinning (`nodeName`)
2. Node labels + `nodeSelector`
3. Node Affinity (hard and soft rules)
4. Taints & tolerations
5. Pod affinity / anti-affinity
6. How to choose the right method

This document explains each of them clearly with examples.

---

# 1. Hard Pinning Using `nodeName`

### **Purpose**

Force the Pod to run on a specific node by name. This bypasses the Kubernetes scheduler.

### **How It Works**

* Kubernetes skips the scheduler entirely.
* The Kubelet on the specified node tries to run the Pod.
* If the node is unavailable or insufficient, the Pod stays Pending.

### **YAML Example**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pinned-pod
spec:
  nodeName: worker-node-1
  containers:
    - name: app
      image: nginx
```

### **Pros**

* Very simple.
* Useful for debugging or testing.

### **Cons**

* Not dynamic or flexible.
* No failover if the node becomes unavailable.
* Not recommended for production workloads.

---

# 2. Node Labels + `nodeSelector`

### **Purpose**

Assign workloads to specific types of nodes using labels.

### **How It Works**

1. You label nodes.
2. Pods request nodes with matching labels.

### **Step 1: Label a Node**

```bash
kubectl label node worker-node-1 node-role=my-special-node
```

### **Step 2: Pod Uses `nodeSelector`**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-selector-pod
spec:
  nodeSelector:
    node-role: my-special-node
  containers:
    - name: app
      image: nginx
```

### **Pros**

* Simple and predictable.
* Good for basic clustering: GPU nodes, SSD nodes, etc.

### **Cons**

* Limited to equality matches (`key: value`).
* Cannot express complex rules.

---

# 3. Node Affinity

Node Affinity is a more powerful and flexible alternative to `nodeSelector`. It supports multiple rules and logical expressions.

There are two types of node affinity:

* **requiredDuringSchedulingIgnoredDuringExecution** (hard requirement)
* **preferredDuringSchedulingIgnoredDuringExecution** (soft preference)

---

## 3.1 Hard Node Affinity

### **Purpose**

Define rules that the scheduler *must* follow.

### **YAML Example**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: affinity-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: node-role
                operator: In
                values:
                  - my-special-node
  containers:
    - name: app
      image: nginx
```

### **Common Operators**

* `In`
* `NotIn`
* `Exists`
* `DoesNotExist`
* `Gt`, `Lt`

### **Use Case Examples**
We are telling the Kubernetes scheduler to
* Must use SSD-backed nodes.
* Must use nodes in a specific region.
* Must run only on GPU-enabled nodes.

---

## 3.2 Soft Node Affinity

### **Purpose**

Tell the scheduler which nodes you prefer, but allow others if necessary.

### **YAML Example**

```yaml
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          preference:
            matchExpressions:
              - key: zone
                operator: In
                values: [zone-a]
```

### **Use Cases**

* Prefer local zone for better latency.
* Spread workloads across zones with weighted preferences.

---

## 3.3 Combining Hard and Soft Affinity

### **Example**

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: disktype
                operator: In
                values: [ssd]
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 50
          preference:
            matchExpressions:
              - key: zone
                operator: In
                values: [zone-a]
```

This means:

* Must run on SSD nodes.
* Prefer zone A if available.

---

# 4. Taints & Tolerations

### **Purpose**

Taints and tolerations control **which Pods are allowed** to be scheduled onto specific nodes. They are essential for workload isolation, dedicating nodes, protecting system-critical nodes, and preventing accidental scheduling.

They work alongside **node affinity**:

* **Node Affinity:** Pod says → “I *want* to run on nodes with these labels.”
* **Taints & Tolerations:** Node says → “Do **NOT** schedule pods here unless they tolerate my taint.”

---

## 4.1 What Are Taints?

Taints are applied **on nodes** and act as rules that **repel Pods** that do not have a matching toleration.

Example taint:

```
dedicated=api:NoSchedule
```

Meaning:

> “Do NOT schedule non-tolerating Pods onto this node.”

### **Taint Format**

```
key=value:effect
```

### **Components**

#### **key**

Defines the **category** or purpose of the taint.
Examples:

* dedicated
* accelerator
* env
* node-type

#### **value**

Specifies a **classification** for the key.
Examples:

* dedicated=api
* accelerator=gpu
* env=prod

#### **effect**

Describes what happens if a Pod does **not** tolerate the taint:

* **NoSchedule** — New Pod is blocked unless tolerated.
* **PreferNoSchedule** — Scheduler avoids the node when possible.
* **NoExecute** — Blocks new Pods and evicts existing Pods that do not tolerate the taint.

---

## 4.2 Applying a Taint to a Node

```bash
kubectl taint nodes worker-node-1 dedicated=api:NoSchedule
```

Meaning:

* Only Pods with a matching toleration may be scheduled on this node.

---

## 4.3 What Are Tolerations?

Tolerations are applied **on Pods** and allow (but do not force) a Pod to be scheduled onto a tainted node.

Example toleration:

```yaml
tolerations:
  - key: "dedicated"
    operator: "Equal"
    value: "api"
    effect: "NoSchedule"
```

This Pod is now **allowed** to run on a node with the taint:

```
dedicated=api:NoSchedule
```

---

## 4.4 Pod Example With Toleration

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: tainted-pod
spec:
  tolerations:
    - key: "dedicated"
      operator: "Equal"
      value: "api"
      effect: "NoSchedule"
  containers:
    - name: app
      image: nginx
```

---

## 4.5 Combining Labels, Taints, and Node Affinity

This is the **most robust and safest pattern** for controlling node selection.

### **Why combine all three?**

| Mechanism         | Purpose                                           |
| ----------------- | ------------------------------------------------- |
| **Label**         | Identifies the node (e.g., GPU node)              |
| **Taint**         | Blocks non‑GPU Pods from landing on this node     |
| **Node Affinity** | Ensures the Pod *selects* this node intentionally |
| **Toleration**    | Allows the Pod to bypass the taint                |

### **Step-by-Step GPU Example**

#### **1. Label the node**

```bash
kubectl label node gpu-node-1 accelerator=gpu
```

#### **2. Taint the node**

```bash
kubectl taint nodes gpu-node-1 accelerator=gpu:NoSchedule
```

Meaning:

* Only Pods that **tolerate** this taint AND
* Also **select the node**

can run here.

#### **3. Pod with Node Affinity + Toleration**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-app
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: accelerator
                operator: In
                values: [gpu]
  tolerations:
    - key: "accelerator"
      operator: "Equal"
      value: "gpu"
      effect: "NoSchedule"
  containers:
    - name: app
      image: my-gpu-image
```

---

## 4.6 Summary

| Concept        | Applied To         | Meaning                                                |
| -------------- | ------------------ | ------------------------------------------------------ |
| **Taint**      | Node               | Repels Pods without matching toleration                |
| **Toleration** | Pod                | Allows Pod to run on a tainted node                    |
| **key**        | Taint & Toleration | Category (e.g., dedicated, accelerator)                |
| **value**      | Taint & Toleration | Classification (e.g., api, gpu)                        |
| **effect**     | Taint              | What scheduler should do (NoSchedule, NoExecute, etc.) |

---

# 5. Pod Affinity & Anti-Affinity

Instead of defining node rules, you define pod-to-pod placement rules.

---

## 5.1 Pod Anti-Affinity — Spread Replicas

```yaml
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: [api]
          topologyKey: kubernetes.io/hostname
```

Meaning:

* Do not place Pods with `app=api` on the same node.
* Ensures high availability.

---

## 5.2 Pod Affinity — Co-Locate with Other Pods

```yaml
spec:
  affinity:
    podAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchExpressions:
                - key: app
                  operator: In
                  values: [redis]
            topologyKey: kubernetes.io/hostname
```

Meaning:

* Prefer nodes already running `app=redis` Pods.
* Helpful for caching or communication-heavy workloads.

---

# 6. Choosing the Right Scheduling Mechanism

Use the following guide:

### **1. Pin a Pod to an exact node**

→ Use `nodeName`

* Good for testing only.

### **2. Simple scheduling to specific node groups**

→ Use Node Labels + `nodeSelector`

* Best for beginner-level cluster segmentation.

### **3. Complex rules (hard + soft constraints)**

→ Use Node Affinity

* Most flexible and recommended for production.

### **4. Reserve nodes for specific workloads**

→ Use Taints & Tolerations

* Perfect for node pools dedicated to system or critical workloads.

### **5. Spread or co-locate Pods relative to each other**

→ Use Pod Affinity / Anti-Affinity

* Ideal for HA and performance-sensitive applications.

### **6. Strictly isolate special nodes for specific workloads**

→ Use Combined Labels + Taints + Node Affinity

* Ensures only intended workloads can run on special node types (e.g., GPU nodes).
* Provides the strongest scheduling guarantees in Kubernetes.

---

# Summary

This document covers the full set of Kubernetes scheduling features. For production environments, the recommended approach is:

* Label your nodes properly.
* Use **Node Affinity** for rules.
* Use **Taints & Tolerations** to protect special nodes.
* Use **Affinity/Anti-Affinity** to optimize workload distribution.
* Use Combined Labels + Taints + Node Affinity for strict node isolation and guaranteed scheduling for special workloads.

This ensures flexible, maintainable, and resilient Pod scheduling.

# Kubernetes Node Maintenance Guide

This guide explains **how to safely perform maintenance on Kubernetes nodes** without causing downtime for your applications. It covers the real-world workflow companies use in production, including cordoning, draining, updating, and restoring a node.

---

## 1. Why Node Maintenance Is Important

Nodes require maintenance for tasks like:

* OS or kernel updates
* Security patches
* Kubelet or container runtime upgrades
* Reboots
* Hardware fixes

To avoid application downtime, Kubernetes provides built‑in tools that let you prepare a node for maintenance **without interrupting running workloads**.

The industry-standard approach is:

**cordon → drain → perform maintenance → uncordon**

---

## 2. Make a Node Unschedulable (`cordon`)

### Command

```bash
kubectl cordon <node-name>
```

### What It Does

* Marks the node as **unschedulable**.
* Prevents **new pods** from being placed on the node.
* Existing pods **continue running normally**.

### When Companies Use It

* Before draining a node
* During investigations
* Short-lived maintenance where existing pods can stay

---

## 3. Evict Workloads Safely (`drain`)

If the maintenance will cause disruption (like a reboot), the node should be **drained**.

### Command

```bash
kubectl drain <node-name> \
  --ignore-daemonsets \
  --delete-emptydir-data
```

### What It Does

* Gracefully evicts **all regular pods** from the node
* Reschedules workloads to healthy nodes
* Respects **Pod Disruption Budgets (PDBs)**
* Skips DaemonSet pods (they recreate automatically later)
* Ensures **zero downtime** for your applications (assuming multiple replicas)

### When Companies Use It

* OS upgrades
* Kernel patches
* Kubelet or container runtime upgrades
* Reboots
* Cloud node image upgrades (EKS/AKS/GKE)

---

## 4. Perform Maintenance

Once the node is drained and empty, maintenance is safe to perform.

### Typical Maintenance Tasks

* Rebooting the machine
* Applying OS or kernel patches
* Upgrading kubelet / CRI-O / containerd
* Updating drivers
* Fixing node-level configuration
* Applying cloud provider updates

During this time, your workloads continue running on other nodes.

---

## 5. Bring the Node Back (`uncordon`)

When the node is fully operational after maintenance:

### Command

```bash
kubectl uncordon <node-name>
```

### What It Does

* Marks the node as **schedulable** again
* Scheduler can place new Pods onto the node

---

## 6. Full Zero-Downtime Workflow

Here is the exact workflow used by real companies:

```bash
NODE=<node-name>

kubectl cordon $NODE
kubectl drain $NODE --ignore-daemonsets --delete-emptydir-data
# Perform your maintenance (patch, upgrade, reboot)
kubectl uncordon $NODE
```

This ensures:
✔ No downtime
✔ Safe pod eviction
✔ Predictable scheduling
✔ Proper node recovery

---

## 7. Maintenance vs Taints — What’s the Difference?

### Cordon / Drain

* Used for **temporary maintenance**
* Operational tasks
* Node becomes safe to update
* Scheduler behavior returns to normal afterward

### Taints & Tolerations

* Used for **long-term scheduling control** (e.g., GPU-only nodes, production-only nodes)
* Not intended for maintenance

| Feature       | Cordon              | Drain            | Taint                    |
| ------------- | ------------------- | ---------------- | ------------------------ |
| Purpose       | Stop new scheduling | Move pods safely | Long-term node isolation |
| Existing Pods | Stay                | Evicted          | Depends on toleration    |
| New Pods      | Blocked             | Blocked          | Blocked unless tolerated |
| Use Case      | Maintenance         | Maintenance      | Workload control         |

---

## 8. How Real Companies Handle Node Maintenance

Most organizations follow exactly this procedure. They may automate it using:

* Cluster Autoscaler
* Karpenter
* EKS Managed Node Groups
* AKS Node Updates
* GKE Node Auto-Repair
* ArgoCD or CI/CD pipelines

But behind the scenes, they all still use:

**cordon → drain → update → uncordon**

This guarantees safe, zero-downtime node maintenance.

---

## 9. Quick Summary

* **cordon** → Stop new pods from scheduling
* **drain** → Safely move existing pods away
* **maintenance** → Update or reboot the node
* **uncordon** → Bring the node back into service

This is the industry standard for safe production maintenance.

# EKS Upgrade Guide (In-Depth, Real-World Explanation)

This document explains **how real companies upgrade Amazon EKS clusters** safely and with zero downtime.

EKS upgrades happen in **two independent parts**:

1. **Control Plane Upgrade** → AWS handles this
2. **Worker Node Upgrade** → companies must handle this carefully

Both steps are required for a complete upgrade.

---

# 1. Understanding EKS Upgrades

AWS automatically manages the **EKS control plane**, but you are responsible for managing and upgrading **worker nodes**.

Nodes do NOT update automatically.

This means:

* Your cluster version may say “1.29”, but your nodes might still be running “1.28”
* Applications continue to run but must eventually be moved to new nodes

Real companies treat node upgrades as **critical maintenance** requiring a controlled, safe process.

---

# 2. Control Plane Upgrade (AWS-Managed)

This is the simplest step.

### How companies do it:

* Go to AWS Console → EKS → Cluster → Upgrade
* Or use CLI: `aws eks update-cluster-version`

### What happens:

✔ AWS upgrades API server, controllers, scheduler
✔ Your workloads continue running
✔ No downtime occurs
✔ Takes ~15–30 minutes

Because AWS runs the control plane, it is resilient and redundant.

---

# 3. Worker Node Upgrade (THE IMPORTANT PART)

Upgrading nodes is where companies need to be careful.

Nodes **must be replaced**, not modified in place.

This ensures:

* Zero downtime
* Safe eviction of pods
* Clean environment with updated AMIs

Companies follow **three main upgrade strategies**.

---

# 4. Approach A: Managed Node Groups (Most Common)

This is the preferred method for 90% of organizations.

### Why?

AWS automates the entire node upgrade process:

* Creates new nodes with the new version
* Cordons old nodes (no new pods)
* Drains old nodes (evicts pods safely)
* Moves pods to new nodes
* Terminates old nodes

### Result:

✔ Zero downtime
✔ No manual steps
✔ No risk of pod loss
✔ Safe node rollout

This is the method used by most modern production teams.

---

# 5. Approach B: Self-Managed Nodes (ASG-Based)

Used mostly by older clusters or companies with special networking/custom setups.

Companies must handle upgrading manually.

### Real-World Workflow:

1. **Create a new Launch Template** with new EKS AMI
2. **Scale up** new nodes in the Auto Scaling Group
3. **Cordon** old nodes

   ```bash
   kubectl cordon <node>
   ```
4. **Drain** old nodes

   ```bash
   kubectl drain <node> --ignore-daemonsets --delete-emptydir-data
   ```
5. **Terminate** old nodes from the ASG
6. **Verify** workloads are healthy on new nodes

This process is used by companies like:

* Lyft
* Stripe
* Expedia
* Any organization running custom AMIs

---

# 6. Approach C: Karpenter-Powered Clusters

Many modern companies use **Karpenter**, which makes upgrades even easier.

### How Upgrade Works with Karpenter:

* Update NodePool with new AMI or Kubernetes version
* Karpenter launches new nodes automatically
* Karpenter evicts workloads from old nodes
* Karpenter deletes old nodes

### Benefits:

* Fastest upgrade method
* Minimal manual work
* Automatic scaling during rollout

---

# 7. Important Real-World Rules

## 💡 Rule 1: Companies **never** upgrade nodes in-place

No one logs into the node and types:

* `yum update`
* `apt upgrade`
* `kubelet upgrade`

This is too risky.

Instead:
**Nodes are replaced with new ones.**

---

## 💡 Rule 2: Zero downtime depends on drain behavior

Every node replacement follows:

```
cordon → drain → replace node → uncordon (for new node)
```

This ensures pods move safely.

---

# 8. Which Method Do Companies Choose?

| Environment Type          | Real-World Method                 |
| ------------------------- | --------------------------------- |
| Modern cloud-native teams | **EKS Managed Node Groups**       |
| High-scale, dynamic       | **Karpenter**                     |
| Legacy setups             | Self-managed ASG nodes            |
| Regulated industries      | Blue/Green node groups for safety |

---

# 9. Visual Summary

### **EKS Upgrade Flow**

```
1. Upgrade Control Plane (AWS manages this)
2. Upgrade Worker Nodes:
   - Managed Node Group → Auto replace nodes
   - Self-Managed Nodes → Manually replace nodes
   - Karpenter → Automatically recreate nodes
```

---

# 10. Key Takeaways

* EKS control plane upgrades are **automatic and zero-downtime**
* Worker nodes always require **replacement**, not patching
* Real companies rely on:

  * Managed Node Groups
  * Karpenter
  * Or controlled ASG rollouts
* Every node upgrade uses the core workflow:

  ```
  cordon → drain → replace node → uncordon
  ```
* This is the safest and most widely used real-world process
