## Using a GoDaddy Domain with AWS (Route 53, ACM, and Your Application)

This document explains, step by step, how a domain purchased from **GoDaddy** can be connected to AWS and used securely with your application.

The goal is simple:

* Buy a domain
* Let AWS manage DNS
* Secure the domain with HTTPS
* Use it with your application (ALB / Ingress)

---

## 1. Purchasing a Domain from GoDaddy

The first step is owning a domain name (for example: `example.com`).

When you purchase a domain from GoDaddy:

* You become the domain owner
* GoDaddy acts as the **domain registrar**
* DNS can be managed either by GoDaddy or by another DNS provider

At this stage, the domain does **nothing** yet — it only exists.

---

## 2. Understanding DNS and Why Route 53 Is Used

DNS is what connects a domain name to an actual resource like a Load Balancer or server.

AWS Route 53 is commonly used because:

* It integrates cleanly with AWS services
* It works seamlessly with Load Balancers and certificates
* It supports automation and scalability

Even if the domain is bought from GoDaddy, **DNS can be managed by Route 53**.

---

## 3. Creating a Hosted Zone in Route 53

A **Hosted Zone** is AWS’s representation of your domain’s DNS records.

When you create a public hosted zone in Route 53:

* AWS generates a set of **Name Servers (NS records)**
* These name servers tell the internet where DNS queries should go

At this point:

* Route 53 knows about your domain
* The internet does not (yet)

---

## 4. Pointing GoDaddy to Route 53

To make Route 53 active, GoDaddy must delegate DNS control to AWS.

This is done by updating **Name Servers** in GoDaddy:

* GoDaddy provides default name servers
* You replace them with the Route 53 name servers

Once updated:

* GoDaddy remains the registrar
* Route 53 becomes the DNS manager

This change may take some time to fully propagate.

---

## 5. Adding DNS Records in Route 53

After delegation is complete, all DNS records are managed in Route 53.

Common records include:

* **A / Alias records** pointing to an Application Load Balancer
* Records for subdomains such as `app.example.com`

From this point forward, GoDaddy is no longer involved in daily DNS changes.

---

## 6. Creating an SSL Certificate in AWS Certificate Manager (ACM)

To enable HTTPS, an SSL/TLS certificate is required.

AWS Certificate Manager (ACM) is used to:

* Create certificates at no additional cost
* Automatically renew certificates

When requesting a certificate:

* You specify the domain (e.g. `example.com`, `*.example.com`)
* AWS asks you to prove domain ownership

---

## 7. DNS Validation for the Certificate

The recommended validation method is **DNS validation**.

ACM provides a special DNS record that must be added to Route 53.

Because Route 53 manages DNS:

* AWS can automatically create the validation record
* Certificate validation completes quickly

Once validated:

* The certificate becomes **Issued**
* It is ready to be used by AWS services

---

## 8. Using the Certificate with Your Application

The issued certificate is attached to:

* An Application Load Balancer
* Or an Ingress backed by an ALB

This allows:

* HTTPS (port 443)
* Secure traffic to your application

The application itself does not need to manage certificates — AWS handles it.

---

## 9. End-to-End Flow (Big Picture)

1. Domain is purchased from GoDaddy
2. Route 53 hosted zone is created
3. GoDaddy name servers are updated to Route 53
4. DNS records are managed in Route 53
5. SSL certificate is created and validated in ACM
6. Certificate is attached to the Load Balancer
7. Users access the application securely using HTTPS

---

## Final Takeaway

* GoDaddy is used only to **own the domain**
* Route 53 is used to **manage DNS**
* ACM is used to **secure the domain with HTTPS**
* The Load Balancer connects the domain to your application

Each service has a clear role, and together they form a clean, production-ready setup.
