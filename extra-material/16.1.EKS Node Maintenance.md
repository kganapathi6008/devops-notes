# EKS Node Maintenance with Cluster Autoscaler and Karpenter

This document explains **exactly how node maintenance works** in EKS when using **Cluster Autoscaler** or **Karpenter**, focusing on **where AMI updates happen, where node deletion happens, and what gets replaced vs patched**.

The key principle throughout this document is **immutable infrastructure**:

> Nodes are **never patched in place**. They are **drained, deleted, and replaced**.

---

## Cluster Autoscaler + Node Maintenance

### Step-by-step behavior (precise and real-world)

1. You update the **AMI** in the **Launch Template** or **Managed Node Group**.

   * This is the **only place** where a new AMI is selected.
   * Cluster Autoscaler is **not involved** in this step.

2. The AMI update affects **only future node launches**.

   * Existing nodes keep running with their **current AMI**.
   * No OS patching occurs.
   * No kubelet upgrade occurs.
   * No reboot occurs.
   * No files on the existing node change.

3. Kubernetes sees **no difference** on existing nodes.

   * Cluster Autoscaler takes **no action** at this point.

4. You select a node that requires maintenance.

5. You **cordon** the node.

   * New pods are prevented from scheduling on it.

6. You **drain** the node.

   * Pods are evicted safely and rescheduled elsewhere.

7. You **terminate the node** (manually or via node group rollout).

   * This is where **node deletion actually happens**.

8. Cluster Autoscaler detects **reduced cluster capacity**.

   * "A node is missing."

9. Cluster Autoscaler increases the node group desired capacity.

   * It **does not choose the AMI**.

10. AWS creates a **new node**.

    * The AMI is picked from the **updated Launch Template / Node Group**.
    * Pods reschedule automatically.

**Result:** Zero downtime, old nodes removed, new nodes created with the new AMI.

---

## Karpenter + Node Maintenance

### Step-by-step behavior (precise and real-world)

1. You update the **AMI** in the **NodeClass**.

   * This defines the **desired state** for new nodes.

2. Existing nodes are **not modified**.

   * No OS updates.
   * No kubelet upgrades.
   * No in-place patching.

3. The AMI update affects **only newly created nodes**.

   * Existing nodes continue running unchanged.

4. You select a node that requires maintenance.

5. You **cordon** the node (optional but recommended).

   * Prevents new pods from landing.

6. You **drain** the node.

   * Pods are evicted safely.

7. You **delete / terminate the node**.

   * This is where **node deletion happens**.

8. Karpenter detects **unschedulable pods or missing capacity**.

9. Karpenter creates a **new node directly**.

   * No Auto Scaling Group or Managed Node Group is involved.

10. The new node is launched using the **updated NodeClass AMI**.

    * Pods reschedule automatically.

**Result:** Zero downtime, fast replacement, and fully immutable nodes.

---

## Key Comparison Summary

| Aspect               | Cluster Autoscaler           | Karpenter        |
| -------------------- | ---------------------------- | ---------------- |
| Where AMI is defined | Launch Template / Node Group | NodeClass        |
| Who creates nodes    | AWS (ASG / MNG)              | Karpenter        |
| In-place patching    | Never                        | Never            |
| Node upgrade method  | Drain & replace              | Drain & replace  |
| Maintenance model    | Capacity-based               | Pod-demand-based |

---

## Golden Rules to Remember

* Updating an AMI **never updates existing nodes**.
* Nodes are **always replaced, never patched**.
* Autoscalers **react to capacity loss**, they do not perform upgrades.
* Safe maintenance always follows:

```
cordon → drain → delete → recreate
```

This is the **industry-standard, production-safe approach** used in real EKS clusters.

# Automating EKS Node Maintenance in the Real World

This document explains **how node maintenance is actually automated in production EKS clusters**, based on how real companies operate at scale. It focuses on **automation patterns**, not manual kubectl steps.

The core idea across all approaches:

> **Humans do not babysit nodes. Automation replaces nodes safely and continuously.**

---

## The Foundational Principle: Immutable Nodes

In real-world environments:

* Nodes are **never patched in place**
* OS, kernel, and kubelet updates are done by **replacing nodes**
* Maintenance is a **continuous automation process**, not a one-time task

This principle applies whether teams use **Cluster Autoscaler** or **Karpenter**.

---

## Automation Model 1: Managed Node Groups (Most Common)

### Who uses this

* Majority of EKS customers
* Enterprises, SaaS companies, regulated industries

### How maintenance is automated

1. AWS releases a **new EKS-optimized AMI**.
2. Teams update the **Managed Node Group version** (console, CLI, or IaC).
3. AWS automatically performs a **rolling node replacement**:

   * New nodes are launched with the new AMI
   * Old nodes are cordoned and drained
   * Pods are rescheduled
   * Old nodes are terminated
4. Cluster Autoscaler cooperates by allowing scale-up during the rollout.

### Key characteristics

* No manual cordon or drain commands
* No SSH or patching
* Fully AWS-managed workflow

This is the **default automation path** for most production clusters.

---

## Automation Model 2: Infrastructure as Code + CI/CD (Common in Mature Teams)

### Tools typically used

* Terraform / CloudFormation
* GitHub Actions, GitLab CI, Jenkins
* AWS CLI

### How maintenance works

1. AMI ID is updated in **code** (Launch Template or Node Group).
2. Change is reviewed and merged like application code.
3. CI/CD pipeline applies the change.
4. A **rolling replacement** is triggered:

   * Nodes are drained automatically
   * New nodes come up
   * Old nodes are terminated
5. Observability checks validate rollout success.

### Why companies prefer this

* Auditable changes
* Repeatable process
* Safe rollbacks

---

## Automation Model 3: Karpenter (Modern, Cloud-Native Teams)

### Who uses this

* High-growth startups
* Platform teams
* Cost-optimized environments

### How maintenance is automated

1. Team updates the **NodeClass AMI selector**.
2. No immediate action is taken on existing nodes.
3. Automation (or consolidation policies) gradually removes old nodes:

   * Nodes are deleted intentionally
   * Or Karpenter consolidates them
4. Karpenter creates **new nodes with the updated AMI**.
5. Workloads shift automatically.

### Key characteristics

* No node groups
* No ASGs
* No rolling updates
* Replacement is demand-driven

This is the **closest model to "hands-off" maintenance**.

---

## Automation Model 4: Scheduled Maintenance (Large Enterprises)

### Common in

* Banks
* Healthcare
* Government

### How it works

1. Maintenance window is defined (weekly/monthly).
2. Automation selects nodes in batches.
3. For each batch:

   * Nodes are cordoned
   * Nodes are drained
   * Nodes are terminated
4. Autoscaler restores capacity.
5. Health checks gate the next batch.

This ensures **predictable, compliant maintenance**.

---

## What Humans Actually Do

In real production environments, humans typically:

* Approve AMI updates
* Monitor dashboards
* Respond to alerts
* Roll back if needed

They **do not**:

* SSH into nodes
* Patch nodes manually
* Reboot nodes by hand

---

## The Universal Automation Loop

Regardless of tooling, every real-world setup follows this loop:

```
Build new AMI → Update desired state → Replace nodes → Verify → Repeat
```

Autoscalers only ensure **capacity and safety** during this process.

---

## Final Reality Check

If your process involves:

* Logging into nodes
* Running OS updates manually
* Maintaining nodes long-term

Then it is **not** how modern EKS clusters are operated.

Real-world EKS maintenance is:

* Automated
* Immutable
* Auditable
* Zero-downtime

That is how companies actually run Kubernetes at scale.
