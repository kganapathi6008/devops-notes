
# Memory-Based Autoscaling (HPA) + OOMKilled Demonstration

## Part A — Horizontal Pod Autoscaler (HPA) for **Memory**

### 1. Overview

Kubernetes HPA can scale pods based on **memory utilization**. Memory-based autoscaling is helpful when:

* The application is memory‑intensive
* CPU is not the bottleneck
* You want scaling to react to RAM pressure

HPA **increases or decreases pod replica count** based on:

* Percentage of **memory usage** relative to **memory requests**

⚠️ **Important:** HPA does *not* resize pods. It only changes the number of pods.

---

### 2. Requirements

To use memory-based autoscaling, you must have:

* **Metrics Server** installed (`kubectl top pods` must work)
* Kubernetes supporting `autoscaling/v2`
* Pods with **memory requests** set (HPA uses % of memory requests)

---

### 3. Example: Deployment with Memory Requests & Limits

Below is a Deployment configured *only for memory demonstration*. CPU is not used or referenced.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: memory-app
spec:
  selector:
    matchLabels:
      app: memory-app
  template:
    metadata:
      labels:
        app: memory-app
    spec:
      containers:
      - name: memory-app
        image: registry.k8s.io/hpa-example
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "256Mi"   # HPA % is calculated from this value
          limits:
            memory: "512Mi"
---
apiVersion: v1
kind: Service
metadata:
  name: memory-app
spec:
  selector:
    app: memory-app
  ports:
  - port: 80
```

---

### 4. HPA for **Memory-Based Autoscaling Only**

Memory-only HPA example:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: memory-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: memory-app
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 75     # target = 75% of 256Mi (~192Mi per pod)
```

#### How this works

If any pod uses **more than 75%** of its memory *request* (256Mi), i.e., above ~192Mi,
HPA will **scale up the Deployment**.

When memory drops below the target for a sustained period, HPA scales down.

---

### 5. Watch HPA Behavior

```
kubectl get hpa memory-app-hpa --watch
```

Example output:

```
NAME             TARGET         MINPODS   MAXPODS   REPLICAS
memory-app-hpa   140%/75%       1         10        4
```

This means:

* Pods are using 140% *of their memory request* (~358Mi)
* HPA increased replicas to 4

## Part B — **OOMKilled Demonstration**

This demo intentionally causes the container to exceed its **memory limit**, triggering an **OOMKilled** event.

This is ideal for teaching resource isolation and cgroup enforcement.

---

### 1. App Code — Memory Allocation Example (Flask)

```python
# app.py
from flask import Flask
app = Flask(__name__)

memory_blocks = []

@app.route('/')
def index():
    return "Memory OOM test running
"

@app.route('/alloc/<int:mb>')
def alloc(mb):
    # allocate MB into memory_blocks list
    block = bytearray(mb * 1024 * 1024)
    memory_blocks.append(block)
    return f"Allocated {mb}MiB. Total blocks: {len(memory_blocks)}
"

@app.route('/killme')
def killme():
    # allocate 300Mi to force OOM
    block = bytearray(300 * 1024 * 1024)
    memory_blocks.append(block)
    return "Attempted 300Mi allocation
"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

---

### 2. Dockerfile

### 3. Deployment to Trigger OOMKilled

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: oom-test
spec:
  selector:
    matchLabels:
      app: oom-test
  template:
    metadata:
      labels:
        app: oom-test
    spec:
      containers:
      - name: memory-hog
        image: yourusername/oom-test-app:latest  # Replace with your repo
        ports:
        - containerPort: 5000
        resources:
          requests:
            memory: "256Mi"
          limits:
            memory: "256Mi"   # exceeding this triggers OOMKilled
---
apiVersion: v1
kind: Service
metadata:
  name: oom-test-service
spec:
  selector:
    app: oom-test
  ports:
    - port: 80
      targetPort: 5000
```

---

### 4. Run the Demo

#### Deploy

```
kubectl apply -f oom-test.yaml
POD=$(kubectl get pods -l app=oom-test -o jsonpath='{.items[0].metadata.name}')
```

#### Check pod

```
kubectl get pod $POD
```

#### Trigger OOM

```
kubectl exec -it $POD -- curl http://localhost:5000/killme
```

This forces a 300Mi allocation → exceeds 256Mi limit → container killed.

#### Watch pod restart

```
kubectl get pods -w -l app=oom-test
```

Expected:

```
0/1   OOMKilled   1
1/1   Running     2
```

#### Confirm OOMKilled

```
kubectl describe pod $POD
```

Look for:

```
Last State:  Terminated
Reason:      OOMKilled
Exit Code:   137
```

## Part C — HPA + OOMKilled Relationship

* HPA scales **number of pods**, not memory size.
* A pod can autoscale up to many replicas but still OOMKill individually if memory **limit** is too small.
* Memory autoscaling does NOT prevent OOMKilled events.
* If all pods get OOMKilled under load → consider:

  * increasing `limits.memory`
  * using **Vertical Pod Autoscaler** (VPA)
  * analyzing memory leaks in the application
